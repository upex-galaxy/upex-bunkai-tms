# Business Feature Map Generator

> **Purpose**: Generate or update `.context/business/business-feature-map.md` — a comprehensive inventory of every feature in the system, so developers can plan implementation, extension, and refactor work against a single source of truth.
> **Output**: `.context/business/business-feature-map.md` (created if missing).
> **Audience**: Developers planning roadmap, scoping new work, or onboarding to an unfamiliar module.

**Target**: $ARGUMENTS (module filter, domain name, or leave blank for full system)

---

## What this produces

A single document that catalogs **every feature** of the system with:

- Feature identification, status, and maturity (Stable / Beta / Planned / WIP / Deprecated)
- CRUD matrix per entity (which ops are implemented end-to-end)
- API endpoint inventory grouped by domain
- UI component / page inventory (forms, dashboards, modals)
- Third-party integrations (payments, email, auth, analytics, storage)
- Feature flags + WIP signals (env vars, TODOs, stub handlers, recent branches)
- Discovery gaps (suspected features without confirmation, deprecated code paths)

This is the **feature-centric** complement to:

- `business-data-map.md` (data-centric — entities, flows, ownership)
- `business-api-map.md` (API-centric — endpoint contracts, auth, error shapes)

Together they let `/master-implementation-plan` and `/sprint-development` reason about scope without re-reading the codebase from scratch.

## Sources (use ALL available)

Exhaust every source. Do not rely on code alone — cross-reference with DB, API, and existing docs.

| Source                   | What to extract                                                             | Tool                                                                                               |
| ------------------------ | --------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| API routes / endpoints   | Features exposed via HTTP — each endpoint is a developer-visible capability | Read route files; if OpenAPI exists, parse `api/openapi.json`; use `[API_TOOL]` if available       |
| Frontend routes + pages  | User-facing features — each page/form/modal is a user operation             | Read `{{FRONTEND_REPO}}/{{FRONTEND_ENTRY}}` — focus on routes, pages, forms, modals, dashboards    |
| Database schema          | Entities that back features — CRUD capabilities per entity                  | `[DB_TOOL]` (Supabase MCP) — read-only queries for table structure and FK relationships            |
| Backend services         | Business logic, validation, processing, background jobs                     | Read `{{BACKEND_REPO}}/{{BACKEND_ENTRY}}` — focus on services, controllers, handlers, workers      |
| Package dependencies     | Third-party integrations (payments, email, auth, analytics, monitoring)     | Read `package.json`, `requirements.txt`, `Gemfile`, etc.                                           |
| Feature flags / env vars | Disabled, experimental, or beta features                                    | Grep for `FEATURE_`, `ENABLE_`, `BETA_`, `isEnabled`, `feature.*flag` in codebase + `.env.example` |
| Existing context         | PRD, SRS, business-data-map, domain glossary                                | `.context/PRD/`, `.context/SRS/`, `.context/business/business-data-map.md`                         |
| Git history (recent)     | Recently added, in-progress, or deprecated features                         | `git log --oneline -30` plus `git branch -a` for active feature branches                           |

**Golden rule**: a feature is any **capability the system offers** — API endpoints, UI actions, background processes, integrations. If a user or system can DO it, it's a feature.

## Workflow

### Step 1: Identify scope

- Read `$ARGUMENTS`:
  - If empty → full-system scan
  - If a module/domain name (e.g. `billing`, `auth`) → scope reads to that domain
  - If a path (e.g. `apps/api/src/billing/`) → restrict file reads to that subtree
- Report scope back to the user before doing the heavy lifting.

### Step 2: Read API routes

- Discover the API entrypoint(s): Next.js `app/api/`, Express/Fastify route registration, NestJS controllers, etc.
- For each endpoint extract: `METHOD path | auth-required? | handler file | brief purpose`.
- If `api/openapi.json` exists, prefer it (faster, canonical) and use raw code only to fill gaps.
- Group endpoints by domain (auth, billing, users, products…).

### Step 3: Map frontend pages

For each page, form, modal, dashboard found:

- What user action does it enable?
- What data does it display or collect?
- Which API endpoints does it call? (grep for fetch / axios / api-client usage in the page tree)
- Which user roles can access it? (look for middleware, route guards, RLS hints)

Look for: route segments, form components, modal/dialog components, dashboard widgets, table/list views, primary action buttons.

### Step 4: Cross-check with DB schema and backend services

- Query DB schema via `[DB_TOOL]` (Supabase MCP) — list tables, columns, FK relationships.
- Read backend services / handlers in scope.
- Build the CRUD matrix: for each entity, mark which of C/R/U/D ops have:
  - An API endpoint
  - A UI affordance
  - A backend service / handler
- Flag entities with partial CRUD (read-only, soft-delete only, etc.) — these are common gaps.

### Step 5: Grep for feature flags and env vars

- Grep the codebase for: `FEATURE_`, `ENABLE_`, `BETA_`, `isEnabled(`, `feature.*flag`, `flags\.`.
- Read `.env.example` (and `.env.staging.example` / `.env.production.example` if present) for documented flags.
- For each flag note: name, default value, where it's read, what it gates.

### Step 6: Read recent git history

```bash
git log --oneline -30
git branch -a --sort=-committerdate | head -20
```

Use this to spot:

- Features added in the last ~2 weeks (likely still warm — may be Beta)
- Active feature branches not yet merged (WIP features)
- Recent reverts or deprecations (Deprecated features)

### Step 7: Cross-link with business-data-map.md (soft gate)

If `.context/business/business-data-map.md` exists:

- Every entity in the data map should have CRUD coverage in this map (or be explicitly flagged as read-only / system-owned).
- Every business flow in the data map should map to at least one feature.
- Flag mismatches in the **Discovery gaps** section — do NOT block.

If `business-data-map.md` does NOT exist:

- Continue without it.
- Note in the output header: `> Companion doc business-data-map.md not found — cross-checks skipped.`
- Suggest the user run `/business-data-map` afterwards for full coverage.

### Step 8: Write output

Write `.context/business/business-feature-map.md` using the **Output structure** below.

In UPDATE mode:

1. Generate the new content in memory.
2. Show a diff summary (sections changed, features added/removed, status changes).
3. Ask the user to confirm before overwriting. NEVER auto-overwrite.

### Step 9: Report gaps

Always close with a Discovery gaps section AND a brief CLI report including:

- Total features by status
- CRUD coverage summary
- Integrations count
- Number of WIP / Beta features
- Top 3 discovery gaps that need team clarification

## Output structure

Write `.context/business/business-feature-map.md` with the following sections in order.

### 1. Inventory summary

```markdown
| Category   | Features | Status         |
| ---------- | -------- | -------------- |
| Core       | [count]  | Stable         |
| Secondary  | [count]  | Stable         |
| Beta       | [count]  | Testing        |
| Planned    | [count]  | In Development |
| Deprecated | [count]  | Being removed  |
```

### 2. Feature catalog (by domain)

One section per domain. Each feature:

```markdown
#### Feature: [Name]

| Aspect           | Value                                |
| ---------------- | ------------------------------------ |
| **ID**           | FEAT-NNN                             |
| **Status**       | Stable / Beta / Planned / Deprecated |
| **Endpoints**    | [list of METHOD path]                |
| **UI**           | [components/pages/routes]            |
| **Owners**       | [roles / actors who use it]          |
| **Dependencies** | [services, integrations]             |
| **Evidence**     | [code path(s)]                       |

**Capabilities:**

- [x] Implemented capability
- [ ] Missing or planned capability
```

### 3. CRUD matrix

```markdown
| Entity | Create | Read | Update | Delete  | Evidence   |
| ------ | ------ | ---- | ------ | ------- | ---------- |
| User   | ✅     | ✅   | ✅     | ⚠️ Soft | api/users/ |
```

Legend: ✅ Full, ⚠️ Partial/conditional, ❌ Not available

### 4. API endpoint inventory

Grouped by domain. Columns: `Method | Endpoint | Purpose | Auth | Handler file`.

### 5. UI component inventory

Three sub-tables:

- **Forms**: `Page route | Form name | Submits to (endpoint) | Validation lib`
- **Dashboards / Views**: `Page route | View name | Data source endpoint(s) | Role gating`
- **Actions**: `Component | Type (modal/dialog/confirm) | Trigger | Endpoint called`

### 6. Third-party integrations

| Service | Purpose | Package | Status | Features using it |
| ------- | ------- | ------- | ------ | ----------------- |

### 7. Feature flags and WIP

**Active flags:**

| Flag | Description | Default | Environment | Read by (file) |
| ---- | ----------- | ------- | ----------- | -------------- |

**WIP signals (from git + grep):**

| Signal                 | Evidence                            | Estimated status |
| ---------------------- | ----------------------------------- | ---------------- |
| `feature/UPEX-XYZ-...` | Active branch, last push N days ago | WIP              |
| `TODO: implement X`    | path:line                           | Planned          |

### 8. Discovery gaps

MANDATORY. List features that:

- Could not be verified from code
- Appear partially implemented
- Have unclear ownership or purpose
- Look deprecated but aren't formally marked as such
- Need team clarification before being scoped

Each gap should be actionable: what to ask, who to ask, or what to grep for next.

## Downstream consumers

This file is consumed by:

- `/master-implementation-plan` — uses the feature catalog and CRUD matrix to scope multi-story epics and identify gaps.
- `/business-api-map` — cross-references the endpoint inventory section.
- `/sprint-development` (Planning step) — reads relevant domain section when scoping a story.

Keep section headings stable so downstream consumers can parse them.

---

## Rules / constraints

1. **Read before write**: in UPDATE mode, always read the existing file first and diff. Never blind-overwrite.
2. **No silent deletions**: if a feature disappears between runs, surface it in the diff and explain why (likely deprecation or rename).
3. **Code is ground truth**: when PRD/SRS disagrees with code, code wins. Note the mismatch in **Discovery gaps**.
4. **Respect scope**: if `$ARGUMENTS` restricts scope, do NOT scan outside it — but DO note in the header which domains were skipped.
5. **No coverage / testing claims**: this is a dev-side feature inventory. Do NOT add test coverage matrices, ATC references, or QA-side relevance. Those live in the qa-side counterpart.
6. **Soft gates only**: if `business-data-map.md` is missing, continue and suggest running `/business-data-map` next.
