# Product Backlog Seed — Initial Backlog From PRD/SRS

> **Purpose**: Seed the initial product backlog tree (epics + their foundational stories) in the issue tracker from a fresh PRD/SRS, then materialize a 1:1 **read-only cache** under `.context/PBI/` via the sync script. This is the **first-touch** workflow for a project — it assumes no epics or stories exist yet and that the PRD + SRS have been generated by `/project-foundation`.
> **Use when**: User says "seed the backlog", "create the initial PBI tree", "bootstrap the epics from the PRD", or equivalent.
> **Persona**: Act as Scrum Master + Product Owner. Decompose the MVP into epics, decompose each epic into INVEST-validated foundational stories, create everything **in Jira** incrementally (per-epic), then materialize the local read-only cache with `bun run jira:sync-issues`, and emit a dependency graph + execution-sprint sequence as the closing artifact.
> **No local authoring**: `epic.md`, `story.md`, `epic-tree.md`, and every per-field file are Jira-mirrored read-only caches — NEVER hand-written (`CLAUDE.md` §9; `references/jira-operations.md` → "No local authoring"; `agentic-dev-core/references/acli-integration.md` D9). Author content in Jira (field or `fallback:` comment per `.agents/jira-required.yaml`), then `bun run jira:sync-issues` materializes the cache.

## Custom field resolution — slug-based, never hardcoded

Custom-field IDs vary per workspace (`customfield_10201` in one tenant, `customfield_10141` in another). This skill is workspace-agnostic methodology, so it resolves every field by **slug** at runtime against `.agents/jira-required.yaml` (and the synced `.agents/jira-fields.json` mapping). Numeric IDs never appear in the skill content.

Slugs written by this workflow:

- `{{jira.acceptance_criteria}}` — Gherkin-formatted scenarios (Given / When / Then), every scenario wrapped in a ` ```gherkin ` fenced code block (anti-pattern `I17`). Persona-observable language only — no endpoint paths, HTTP status codes, table names, or framework names (anti-pattern `I15`).
- `{{jira.business_rules_specification}}` — domain rules and validations (boundaries, role gates, retry semantics, audit guarantees). NOT internal algorithms or implementation patterns (anti-pattern `I15`).
- `{{jira.scope}}` — in-scope bullets only (no out-of-scope mixed in). Capabilities the persona gains, not endpoints / tables / services (anti-pattern `I15`).
- `{{jira.out_of_scope}}` — explicit exclusions; complementary to `scope`. Deferred capabilities, not deferred endpoints.
- `{{jira.mockup}}` — design references (Figma URLs etc.).
- `{{jira.workflow}}` — narrative description of a non-trivial flow, written from the persona's POV (not a code-walkthrough). See anti-pattern `I15`.
- `{{jira.weblink}}` — app URL for the story (conditional — only when known with certainty).
- `{{jira.story_points}}` — **OPT-IN ONLY**. Leave EMPTY by default. Populate only when the user explicitly requested estimation in this session. PO/BA do not estimate; the building team does (Design + Dev + Test). When opted-in: Fibonacci (1, 2, 3, 5, 8); 13+ → split. See anti-pattern `I16`.

Field reads/writes route through `[ISSUE_TRACKER_TOOL]`; the owning tool skill resolves the slug → numeric ID before issuing the call. See `references/jira-operations.md` for the routing decision table.

## Summary nomenclature

**Story summary format is `{Feature} | {Action}`** — see the canonical §Story title format in `SKILL.md`
(anti-pattern I20). The `As a … I want to … so that …` sentence is NEVER the summary; it lives only in the
description `## User story` section. `{Feature}` is the abbreviated feature noun (shared by sibling stories
of the same feature); `{Action}` is the `I want to …` clause as a base-form verb phrase (persona + benefit
dropped). When a feature names a product domain entity that collides with agile/QA vocabulary, prefix it
with the project-domain tag `TMS-` (`User Story → TMS-US`, `Module → TMS-Module`, `Test → TMS-Test`, etc.);
cross-cutting features stay plain (`Authentication`, `Jira Import`). English; ≤ ~80 chars.

**NEVER prefix a story summary with `FR-XXX —`.** The Jira issue key is the only identifier that belongs in the summary; the functional-spec reference is methodology metadata and lives in the body.

Place the source-spec reference as the **first line of the story description body**:

```
**Source spec:** FR-XXX
```

Stories that do not map cleanly to a single FR (cross-cutting, infra, technical-debt) omit the line entirely — never invent an FR identifier.

Epic summaries follow the same rule: no `FR-` or `EPIC-` prefix in the summary text; identifiers come from the issue key.

## No content duplication

The story description body and the dedicated custom fields hold **disjoint** content. Acceptance Criteria, Scope, and Out-of-Scope live **exclusively** in their custom fields (`{{jira.acceptance_criteria}}`, `{{jira.scope}}`, `{{jira.out_of_scope}}`); they never appear as H2/H3 sections inside the description.

Canonical description body sections:

1. `**Source spec:** FR-XXX` (first line, optional).
2. `## User story` — As / I want / So that narrative.
3. `## Business rules` — overflow only, when content does not fit in `{{jira.business_rules_specification}}`.
4. `## Workflow` — overflow only, when content does not fit in `{{jira.workflow}}`.
5. `## Definition of done` — closing checklist.
6. Optional blocks: `Mockup`, `Technical notes`.

Full rationale + audit procedure → `references/description-custom-field-dedup.md`.

## Workflow overview — Jira-first → synced read-only cache

The seed runs incrementally per epic to keep token usage bounded and context tight:

1. **Phase 0** — confirm project key from `.agents/project.yaml`.
2. **Phase 1** — draft the full epic tree (planning artifact only, no writes).
3. **Phase 2** — per epic, repeat:
   - 2.1 Create epic via `[ISSUE_TRACKER_TOOL]` → capture real issue key.
   - 2.2 Transition epic to default status (`{{jira.statuses.epic_default}}`, fallback `Planning`).
   - 2.3 Create child stories via `[ISSUE_TRACKER_TOOL]` (INVEST-validated, slug-populated), linked to the epic.
   - 2.4 Transition each story to default status (`{{jira.statuses.story_default}}`, fallback `Shift-Left QA`).
   - 2.5 Materialize the epic + its stories' read-only cache: `bun run jira:sync-issues pull --epic {PROJECT_KEY}-{ISSUE_NUM}`.
   - 2.6 Cross-story Scope overlap check (this epic's children).
4. **Phase 3** — after all epics + stories exist, link dependencies in Jira, then re-sync.
5. **Phase 4** — final phase: derive Execution Sprint sequence from the dependency graph.

Each phase below describes operation semantics only; the owning tool skill (`/acli` primary, MCP fallback, REST last resort per `references/jira-operations.md`) owns the exact command syntax.

## Folder naming — synced read-only cache

The cache under `.context/PBI/` (materialized by `bun run jira:sync-issues`) uses real Jira keys assigned at creation time. Names have four parts:

1. **Fixed**: `EPIC-` or `STORY-` prefix.
2. **Project variable**: `{PROJECT_KEY}` from `.agents/project.yaml` (e.g. `MYM`, `SHOP`, `BLOG`). Uppercase, 2–5 chars, no spaces.
3. **Jira-assigned**: `{ISSUE_NUM}` — the sequential number Jira returns on create. Never controlled by AI or user. No leading zeros.
4. **Domain-inferred**: `{kebab-descriptor}` — 2–4 words in kebab-case, derived from the PRD/SRS vocabulary.

**Epic folders**: `EPIC-{PROJECT_KEY}-{ISSUE_NUM}-{kebab-descriptor}/`

**Story folders**: `STORY-{PROJECT_KEY}-{ISSUE_NUM}-{kebab-descriptor}/`, placed under the parent epic's `stories/` directory.

**Rules**:

- Use kebab-case in the descriptor (lowercase, hyphen-separated).
- IDs never carry leading zeros (`MYM-2`, never `MYM-002`).
- Keep descriptors concise (2–4 words). Long titles go in the issue summary, not the folder name.
- No snake_case, CamelCase, spaces, or special characters beyond hyphens.

## Phase 2: Per-epic creation loop

**Repeat 2.1 → 2.8 for every epic in `epic-tree.md`, in the sequencing order from Phase 1.**

### 2.1 — Create epic in Jira

**Operation**: create an Epic-type issue via `[ISSUE_TRACKER_TOOL]`.

**Routing**: see `references/jira-operations.md` (operation: *create issue*).

**Inputs to the operation**:

- Project: `{PROJECT_KEY}` from `.agents/project.yaml`.
- Issue type: `Epic`.
- Summary: epic title (no `FR-` or `EPIC-` prefix; the key is the identifier).
- Description: 2–3 paragraphs covering scope, business value, and where this epic sits in the Master Sprint roadmap (if `master-implementation-plan.md` is present).
- Priority: `High | Medium | Low`.
- Labels: `mvp` (plus any project-specific labels).

**Rich-text caveat**: the description is an ADF rich-text field. Apply `references/jira-publishing-gotchas.md` rules (no `code` marks inside `**bold**`; one update call per ADF field on existing issues).

**Capture**: the real issue key (e.g. `MYM-13`) returned by the create call. Persist it in working memory for the next steps.

---

### 2.2 — Transition epic to default status

**Operation**: transition the freshly created epic to its default workflow status via `[ISSUE_TRACKER_TOOL]`.

**Routing**: see `references/jira-operations.md` (operation: *transition status*).

**Target status**: `{{jira.statuses.epic_default}}`. If the slug is unset in `.agents/jira-required.yaml`, fall back to the safe default `Planning`.

**Rationale**: Jira's default landing status is rarely the operational starting status. Transitioning eagerly puts the epic in the project's actual planning column so the team's board view is correct from the first second.

**If the transition target does not exist in the workspace's epic workflow**: report the gap to the user, leave the epic in its default landing status, and continue. Never invent a status name.

### 2.3 — Create foundational stories in Jira

**Repeat for every story listed under this epic in `epic-tree.md`.**

**Operation**: create a Story-type issue via `[ISSUE_TRACKER_TOOL]`, linked to the parent epic by epic link.

**Routing**: see `references/jira-operations.md` (operation: *create issue*).

**INVEST validation before each create**:

| Criterion        | Question                                                       |
| ---------------- | -------------------------------------------------------------- |
| **I**ndependent  | Can this story complete without depending on others?           |
| **N**egotiable   | Is there flexibility in the implementation?                    |
| **V**aluable     | Does the "so that" clause carry clear user / business value?   |
| **E**stimable    | Can the team assign story points with the info available?      |
| **S**mall        | Is it ≤ 8 SP? If not, split before creating.                   |
| **T**estable     | Are the acceptance criteria observable and verifiable?         |

If any criterion fails, split or rewrite the story before creating it. Never push an INVEST-failing story to Jira just to keep the count up.

**Inputs to the operation**:

- Project: `{PROJECT_KEY}`.
- Issue type: `Story`.
- Summary: `{Feature} | {Action}` (see §Summary nomenclature). **No `FR-XXX —` prefix; no `As a … so that …` sentence in the summary.**
- Description body: starts with `**Source spec:** FR-XXX` line (if applicable), then a `## User story` section holding the full `As a … I want to … so that …` sentence, then the canonical sections from §"No content duplication" above. No AC / Scope / Out-of-Scope H2 sections.
- Epic Link: the parent epic's real key from Phase 2.1.
- Priority: `High | Medium | Low`.
- Labels: `mvp` (plus project-specific).
- Custom fields (slug-resolved at the tool layer):
  - `{{jira.acceptance_criteria}}` — minimum 3 Gherkin scenarios (1 happy path + 2 edge / error cases), each wrapped in a ` ```gherkin ` fenced code block (anti-pattern `I17`). Persona-observable language only; no endpoint paths, HTTP status codes, table names, or framework names (anti-pattern `I15`).
  - `{{jira.scope}}` — in-scope bullets only. Bullets describe capabilities the persona gains, not endpoints / DB tables / services (anti-pattern `I15`).
  - `{{jira.out_of_scope}}` — explicit exclusions, also in persona-capability voice.
  - `{{jira.story_points}}` — **OPT-IN ONLY**. Leave EMPTY unless the user explicitly requested estimation in this session. When opted-in: Fibonacci (1, 2, 3, 5, 8); 13+ → split. See anti-pattern `I16`.
  - `{{jira.business_rules_specification}}` — optional. Domain rules (boundaries, role gates, retry semantics, audit guarantees) — NOT internal algorithms.
  - `{{jira.mockup}}` — optional.
  - `{{jira.workflow}}` — optional. Narrates the persona's flow, not the code's flow.
  - `{{jira.weblink}}` — conditional; only when the app URL is known with certainty (from `.agents/project.yaml` or explicit user input). When in doubt, leave empty.

**Rich-text caveat**: every custom field above is an ADF rich-text field except `{{jira.story_points}}` (number) and `{{jira.weblink}}` (URL string). Apply `references/jira-publishing-gotchas.md` rules — split the update into one call per ADF field if your tool layer requires it.

**No-invent rule**: AC, Scope, Out-of-Scope, and Business Rules must derive from observable evidence (PRD, SRS, business maps, user journeys, explicit user input). If a field has no source, mark the gap in working memory and leave the slug empty rather than fabricating content.

**Capture**: every story's real issue key, in the order created. The epic's synced `epic.md` child-story list (materialized in 2.5) reflects these keys automatically once the stories are linked to the epic in Jira — never hand-typed.

### 2.4 — Transition each story to default status

**Operation**: for every story created in 2.3, transition to the default story status via `[ISSUE_TRACKER_TOOL]`.

**Routing**: see `references/jira-operations.md` (operation: *transition status*).

**Target status**: `{{jira.statuses.story_default}}`. If the slug is unset, fall back to the safe default `Shift-Left QA`.

**Rationale**: stories landing directly in a workflow's first column hide the deliberate intent that they are refinement-ready. The default story status surfaces that they are pre-refinement / pre-dev work the team must triage.

**If the transition target does not exist**: report the gap, leave the story in its landing status, and continue.

### 2.5 — Materialize the epic + story read-only cache (sync, do NOT hand-write)

**Goal**: materialize the canonical read-only cache for the epic and all its child stories in one pass. The epic + every story already live in Jira (2.1 + 2.3); the sync creates the canonical folders, `epic.md`, every `story.md`, and one Markdown file per non-empty rich-text custom field (`acceptance-criteria.md`, `scope.md`, `out-of-scope.md`, `business-rules.md`, `workflow.md`, `mockup.md`, …). You NEVER hand-create folders or files (`CLAUDE.md` §9; `agentic-dev-core/references/acli-integration.md` D9).

**Operation**:

```
bun run jira:sync-issues pull --epic {PROJECT_KEY}-{ISSUE_NUM}
```

(`{ISSUE_NUM}` = the epic key captured in 2.1.) Canonical layout the sync produces:

```
.context/PBI/epics/EPIC-{PROJECT_KEY}-{ISSUE_NUM}-{kebab-descriptor}/
├── epic.md (+ feature-implementation-plan.md, feature-test-plan.md when populated)
└── stories/
    ├── STORY-{PROJECT_KEY}-{NUM}-{kebab-descriptor}/story.md (+ acceptance-criteria.md, scope.md, out-of-scope.md, …)
    └── ...
```

**Read-back check (not authoring)**: open the synced `epic.md` + `story.md` files and confirm they reflect the Jira issues you created. The `epic.md` child-story list shows the real linked keys. If a story is missing or a field is wrong, the fix is in Jira (correct the Epic Link / parent, or REST PUT the field per `references/jira-operations.md`), then re-run the sync — never edit the local file. AC / Scope / Out-of-Scope are NOT in `epic.md` / `story.md`; they sync into their own per-field files from the Jira custom fields. Cross-link: `references/description-custom-field-dedup.md`.

---

### 2.6 — Cross-story Scope overlap check (this epic's children only)

**Goal**: detect bad slicing where two stories within the same epic share literal Scope bullets — a symptom of overlapping responsibilities that breaks INVEST-I (independence).

**Operation**: read each story's `{{jira.scope}}` from the synced `scope.md` files materialized in 2.5 (do NOT `acli view` — it returns null for custom fields; `agentic-dev-core/references/acli-integration.md` D8). Pairwise-compare the bullet lists across the epic's children.

**Routing**: detailed reads go through `bun run jira:sync-issues` per `references/jira-operations.md` (operation: *read issue with custom fields*).

**Output to user**: for every literal-identical bullet shared between two stories, surface:

```
overlap_alert: <KEY-A> vs <KEY-B> on bullet "<text>"
```

**Do not auto-fix.** Offer the user three resolution options:

1. **Move** — keep the bullet in one owning story; remove from the other.
2. **Extract** — create a new shared-dependency story that owns the bullet; both original stories `depends on` it.
3. **Accept** — leave the duplicate when it is intentional shared context (record the rationale).

Wait for the user's decision per overlap before continuing. Never silently dedup; the overlap is signal that slicing should be revisited.

## Phase 3: Link dependencies in Jira

**Pre-condition**: all epics + their foundational stories from Phase 2 exist in Jira with real keys and are materialized in the synced cache.

**Goal**: publish the `Blocked by` / `Blocks` / `Related` dependencies (surfaced by the active discovery pass below, sourced from PRD/SRS sequencing, master-plan, business-data-map, and explicit author intent) as real Jira issue links so the dependency graph is queryable and the next phase (sprint sequencing) has live data. Once links exist in Jira, a re-sync materializes each story's `## Dependencies` block — those blocks are read-only cache, never hand-authored.

### Phase 3.0 — Active Dependency Discovery (MANDATORY, before any link write)

Anti-pattern `I18` in `SKILL.md` requires an **active** discovery pass before any link is created. Passive "only link if obviously needed" is rejected.

1. Refresh the full backlog tree first — `bun run jira:sync-issues pull` — then read the materialized `.context/PBI/epic-tree.md` (live Jira backlog graph) alongside the Phase 1 planning draft.
2. Read `.context/business/business-data-map.md` if present (entity foreign-key graph).
3. For every epic and story created in Phase 2, query its current Jira link graph via `[ISSUE_TRACKER_TOOL]` so the discovery pass sees any links that may already exist.
4. Build a candidate matrix `(from, to, link_type_slug, source-of-decision)` where `source-of-decision` is one of: `prd-sequencing`, `srs-sequencing`, `master-implementation-plan`, `business-data-map`, `local-declaration`.
5. **Filter noise**: discard candidates whose only justification is a global / infrastructural prerequisite (auth exists, DB exists, framework is wired up, observability is set up). Those are properties of the project as a whole, not of one story over another. Keep ONLY feature-level, observable, explicit dependencies between specific work items.
6. **Heuristic**: would the candidate dependency disappear if we reordered sprints? YES → it is global noise, drop. NO → it is a real feature-level dependency, keep.
7. Surface the filtered matrix to the user and wait for confirmation before any link is written. Never auto-create.

This step runs once before the link-creation loop below.

**Operation**: for each confirmed dependency, create an issue link via `[ISSUE_TRACKER_TOOL]` using the slug-resolved link type (`{{jira.link_types.dependencies}}` for hard ordering; `{{jira.link_types.blocks}}` is interchangeable per the catalog; `{{jira.link_types.relates}}` for informational ties only).

**Routing**: see `references/jira-operations.md` (operation: *create link*) and the full procedure + directionality table in `references/dependency-linking.md`.

**Hard rules** (full list in `references/dependency-linking.md`):

- Never invent dependencies. Sources of truth: PRD/SRS sequencing, master-implementation-plan Master Sprints, business-data-map entity relationships, explicit author intent surfaced during this session (these become Jira issue links here, then sync back into each story's `## Dependencies` block on the next pull). When a dependency is not observable from these inputs, ask the user or skip — never guess.
- Never use the symmetric fallback (`Relates`) for ordering-sensitive dependencies. Symmetric → direction is lost; the topological sort in Phase 4 collapses.
- If `{{jira.link_types.dependencies}}` resolves to its fallback (degraded to `Relates`), report the degradation to the user before proceeding.

**Reporting**: after linking completes, surface a dependency matrix to the user — `from → to → type → source of decision` — so they can review before sprint sequencing runs.

**Re-sync**: once links are published, refresh the cache so each story's `## Dependencies` block reflects the live links — `bun run jira:sync-issues pull`. Never hand-append the links to a `story.md`.

## Phase 4: Sprint sequencing (closing artifact)

**Pre-condition**: Phase 3 complete (all dependency links published in Jira).

**Goal**: derive an Execution Sprint plan (topological sort of the dependency graph) so the team has a concrete answer to "which stories can start in parallel today?". This is an **input artifact to the human sprint-planning meeting** — never a commitment.

**Operation**: run Kahn's algorithm against the `dependencies` (and `blocks`) link graph as described in `references/sprint-sequencing.md`. Cycle detection is mandatory — a cycle halts the algorithm and is reported to the user as a methodology bug to fix before sequencing can complete.

**Routing**: link arrays at scale are read from the synced cache materialized by `bun run jira:sync-issues pull` — see `references/jira-operations.md` (operation: *read issues with custom fields (batch)*).

**Persistence**: ALWAYS write the result to `.context/PBI/sprint-sequence.md` (overwrite on re-run — the artifact is regeneratively derived from the live graph, not appended).

**Hard rules** (full list in `references/sprint-sequencing.md`):

- Only `{{jira.link_types.dependencies}}` and `{{jira.link_types.blocks}}` contribute to the sort. `Relates` is informational, skipped.
- Cycle detection: halt + report; never silently truncate the graph.
- Output is an artifact, not a commitment — frame it as input to the planning meeting.
