1.The fundamental idea

This repository is an AI-driven development boilerplate. It does not ship a backend or a frontend; it ships the instructions, the knowledge layer, and the tooling that an AI agent uses to build a Next.js + Supabase application on top of it. The repo is a starter — you scaffold the actual app inside it via /project-foundation and /project-bootstrap, then drive feature work through /sprint-development.

For the machine to work, three families of moving parts cooperate: Skills orchestrate long, multi-phase AI work; Commands perform one-shot utilities; and Scripts are what you run yourself in the terminal to keep the configuration substrate healthy. All three feed off the same fuel — a system of variables defined in .agents/project.yaml that resolves to your project's specifics at AI session bootstrap.

The whole machine in one sentence

Three kinds of moving parts (Skills · Commands · Scripts) plus one variable substrate (.agents/project.yaml) — that is the entire mental model. The rest is detail.

Read agentic-development-engineering.md for the strategy behind the practice and CONTEXT.md for why the knowledge layer is split the way it is. This page focuses on how the pieces connect — what to invoke, in what order, and what each piece actually does.

Getting started — 3 paths to bootstrap

Pick the entry point that matches your goal. The magic command is the fastest and what 90% of users want; the other two paths exist for specific cases (GitHub-first workflows, or contributing to the boilerplate itself).

Recommended · npm scaffolder

Magic command

bunx create-agentic-dev@latest <your-repo-name>

One command downloads the boilerplate as a tarball (no git history), renames your project, runs bun install, and hands off to the interactive installer.

  1. Tarball download (latest main)
  2. Rewrite package.json + .agents/project.yaml
  3. Fresh git init -b main + initial commit
  4. bun install
  5. bun run setup — gentle-ai (Engram only), MCPs, .env, optional gh repo create

Details: npm package · source + flags

GitHub-first

"Use this template"

Click → clone → bun install && bun run setup

Create your repo on GitHub from day one. Good when you want the remote first and the local checkout second.

  1. Click Use this template → Create a new repository
  2. Pick owner, name, visibility
  3. Clone YOUR new repo locally
  4. bun install && bun run setup
  5. Manually edit package.json name + .agents/project.yaml project.name

Trade-off: keeps upstream git history; you rename manually. The magic command does both steps automatically.

Contributors only

Manual clone

git clone … && bun install && bun run setup

For hacking on the boilerplate itself — skills, installer, scripts, docs. End-users should NOT use this path; the project will keep the upstream name and full git history.

  1. git clone https://github.com/upex-galaxy/agentic-dev-boilerplate.git
  2. cd agentic-dev-boilerplate
  3. bun install
  4. bun run setup

Use this only if you intend to send PRs against the boilerplate itself.

Why the magic command is the default

The boilerplate ships with upstream commit history and a generic project name. The scaffolder scrubs that history and rewrites the name in two files so your repo looks like your project from commit #1. Both other paths require you to do those steps by hand.

Who you're talking to — AI personality

Before you fire your first command, know who is on the other side of the conversation. The AI in this repo has an opinionated, intentional personality — a deliberate mix of speech style, register, and behavioural defaults — so that what you read feels consistent across sessions, agents, and skills.

Short answer: a senior software engineer who reports to a Project Manager. Competent, parsimonious with words, allergic to theatre, biased toward planning over improvisation, and disciplined about translating technical work into business value when speaking to you.

The six strategies you'll notice

Caveman mode

Word compression

Drops articles, fillers, and pleasantries. Fragments OK. Cuts roughly 75% of tokens without losing accuracy. Code blocks stay fully written. Toggle with /caveman lite|full|ultra or disable with stop caveman.

Butler Pattern

Information granularity

Headline first (one line that resolves your literal question), then an atomic bullet menu of related topics. You pull the thread that matters. No paragraphs buried in walls of text.

PM Voice Default

Vocabulary register

Headlines report user/business value, not technical action. Bullets are a SINGLE menu — each bullet's register (value-framed or technical) is chosen per topic, so a file path and a UX-impact statement can sit side by side. In foreground replies, a short attention-priming punch precedes the headline; when a menu has 3+ bullets, a short orientation question invites you to pull a thread. Auto-suspends when you mention a file, command, or library by name.

Background-narrator

Lifecycle signaling

When running as a background job, emits lifecycle signals (result:, needs input:, failed:) so a classifier can track progress without watching live.

Language mirror

Locale

Detects your language and replies in it. Repo artifacts (code, commits, PRs, branch names, Jira issues) stay in English regardless of conversational language.

Visual Mapping Bias

Form

When the content is mappable — comparisons, hierarchies, flows, impact maps — the AI prefers a table, ASCII diagram, tree, box layout, or state machine over a paragraph of prose. The visual replaces prose, it does not decorate alongside it. Can sit below the headline or inside a single bullet. Plain ASCII preferred over Unicode box-drawing when terminal rendering is uncertain.

How to talk to it

  • Speak naturally in your own language. The AI mirrors you — no need to switch to English.
  • Drop a file path, command, or library name in your message if you want a technical reply that turn (PM Voice auto-suspends).
  • State your goal, not your implementation idea if you want the AI to push back when there's a simpler path.
  • Ask for "technical mode" / "speak technically" (in any language) to force a technical reply for the rest of the session, or ask for "PM mode" / "PM voice" to restore the default.
  • Say "stop caveman" / "normal mode" (in any language) to disable token compression and read full-form prose.
Read the full guide before you start. The personality is layered: traits, strategies, override phrases, suspension triggers, and how to evolve it over time. Full reference (with all toggles, examples, and edge cases) lives in docs/ai-personality.md. It mirrors the runtime contract defined in CLAUDE.md sections 1–3.

2.Three categories of moving parts

Everything you can invoke in this repo belongs to one of these three categories. The mental model holds end-to-end:

Type What it is Where it lives Who invokes it
Skill
Multi-file markdown that orchestrates the AI through long, multi-phase tasks (subagents, decisions, references). .claude/skills/<name>/SKILL.md + references/ The AI auto-activates when your prompt matches the skill's triggers. Force-load via /skill-name.
Command
Single-file markdown for one-shot, focused utilities. .claude/commands/<name>.md You explicitly type /<name>. No auto-trigger.
Script
TypeScript CLI that runs in the terminal. scripts/*.ts and cli/*.ts You run bun run <script> directly.

The three categories share the same fuel: the variable substrate in .agents/project.yaml (and Jira metadata in .agents/jira-fields.json + .agents/jira-required.yaml). Skills resolve {{VAR}} and {{jira.*}} placeholders against it on every AI session; commands use the same resolver; scripts read the YAML and write back to it. See §6 Circuit 1 for the full resolution flow.

3.The lifecycle: five phases

Each phase has a clear trigger, a deterministic output and a frequency. You move forward when the previous phase's output exists; you loop back to Maintenance whenever documentation drifts from reality.

Cadence at a glance

Phase Cadence Typical duration Re-run trigger
0 · Bootstrap One-time per clone 15-30 min Fresh clone or env reset
1 · Foundation One-time per product 2-4 hours Major product pivot or architectural rewrite
2 · Scaffold One-time post-Foundation 1-2 hours Brand-new project; not re-run during normal life
3 · Backlog Continuous (weekly+) 30-60 min per refinement New epic, story refinement, AC/Gherkin updates
4 · Sprint Per ticket 1-8 hours Every Jira ticket you implement
5 · Maintenance On drift 15-60 min Doc drift, business-map drift, skill changes

4.Foundation host vs. project scaffolding

Two names sound similar but play different roles. agentic-dev-core is a passive doctrine host loaded by other skills. /project-bootstrap is the workflow skill that scaffolds your codebase. They are NOT interchangeable.

Piece When you invoke it What it does Frequency
/project-bootstrap (skill) Once per project, AFTER /project-foundation finishes. Scaffolds the codebase: backend skeleton (DB, API, types, OpenAPI, auth, env), frontend skeleton (design system from DESIGN.md, routing, project structure). Modifies THIS repo. One-time per project

A passive reference host you never invoke

agentic-dev-core is a passive reference host — it hosts shared doctrine (briefing template, dispatch patterns, orchestration doctrine, skill-composition strategy, behavioral layer, model routing) that workflow skills cite on demand. When /sprint-development, /product-management, /project-foundation, or /project-bootstrap delegates to a subagent, the AI loads agentic-dev-core/references/briefing-template.md, dispatch-patterns.md, and orchestration-doctrine.md automatically. You never invoke agentic-dev-core directly — it has no bootstrap action. Foundation files ship with the cloned repo.

5.The Jira state machine

/sprint-development drives the ticket through a four-state Jira workflow. Every transition is owned by a specific stage of the sprint loop — the AI does not skip ahead. You confirm at the gates.

Each transition is owned by a sprint-development stage. The repo never reaches "Done" — QA lives in the sister repo (agentic-qa-boilerplate).

Transition Owned by Trigger condition
Ready For Dev → In Progress Stage 1 — Planning Implementation plan presented and approved. Feature branch created (e.g. feature/UPEX-123-add-login).
In Progress → In Review Stage 3 — Code Review All commits pushed. PR opened against staging. Review run (single-perspective).
In Review → Ready For QA Stage 4 — Staging Deploy PR merged into staging. Staging deploy green. Smoke check OK. QA can now pick the ticket up.
Mnemonic: Planning opens the ticket, Code Review opens the PR, Staging Deploy opens the door to QA. Production is gated and lives outside the state machine.

A note on the sister repo

This boilerplate is a dev-side machine. Once a ticket reaches Ready For QA, it is the QA team's job to validate it, file bugs, and eventually move it to Closed. The QA workflow lives in the sister repo agentic-qa-boilerplate, with its own Jira state machine (Ready For QA → In Testing → Tested / Closed) and its own skill set (/sprint-testing, /test-automation, etc.). Do not invoke QA-side skills from this repo.

6.How everything connects internally

Two circuits explain the wiring. Circuit 1 is about variable resolution (the static substrate). Circuit 2 is about subagent orchestration (the dynamic execution model).

Circuit 1 — Variables

Variables flow: you fill the YAML once → AI reads it on every session → placeholders resolve to real values before any tool call.

.env vs .agents/project.yaml — two separate systems

Do not conflate them. They have different consumers, different lifecycles, and only one of them is committed.

.env .agents/project.yaml
Purpose Next.js + Supabase runtime secrets and config (also the MCP credentials Claude reads at session start). AI context-engineering variables for {{VAR}} resolution.
Consumers Next.js runtime (next dev, route handlers, server actions), Supabase client, MCP servers (Tavily, Supabase, n8n), test runners (Vitest, Playwright), Atlassian CLI (acli). AI agents (Claude Code, OpenCode, Cursor, Codex) when resolving skill / template / doc references.
Examples LOCAL_USER_EMAIL, STAGING_USER_PASSWORD, SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY, TAVILY_API_KEY, ATLASSIAN_API_TOKEN. PROJECT_KEY, WEB_URL, API_URL, JIRA_URL, DB_PROJECT_REF, default_env.
Secrets? Yes (passwords, tokens, API keys). No — must remain commit-safe.
Committed? Gitignored. .env.example is the committed template. Committed.
Lifecycle Edited per developer / per CI runner. Edited once when adopting the boilerplate; rarely changes after.

The variable substrate has four syntaxes that look similar but resolve from different files. The full contract lives in .agents/README.md; the short version:

  • {{VAR_NAME}} → resolves from .agents/project.yaml (project-level static config). Env-scoped vars like {{WEB_URL}}, {{API_URL}}, {{DB_PROJECT_REF}} resolve against the active environment (default testing.default_env).
  • {{environments.<env>.<var>}} → explicit cross-env reference, useful when a prompt needs to compare values across environments regardless of the active one.
  • <<VAR_NAME>> → session variable, computed at runtime by the calling skill (e.g. <<ISSUE_KEY>> extracted from the git branch). Never persisted.
  • {{jira.<slug>}} → portable Jira custom-field reference, resolves through .agents/jira-required.yaml (manifest) + .agents/jira-fields.json (workspace catalog). Used by /sprint-development and /product-management when they need to read or update fields like acceptance_criteria_gherkin.

Circuit 2 — Orchestration

Each workflow skill declares its own dispatch points; the canonical briefing template + dispatch patterns live in agentic-dev-core/references/ so subagents load them once, fresh.

This is the canonical pattern the workflow skills follow. The orchestrator reads its own dispatch strategy, writes a 6-component briefing for each subagent (Goal · Context docs · Skills to load · Exact instructions · Report format · Rules), and the subagent loads tool skills (/acli, /git-flow-master, /playwright-cli) to actually run shell commands. This is what keeps the main conversation lean — the heavy reading happens inside subagents.

Circuit 3 — Skill-before-Bash auto-load

A third circuit governs shell-out: when Bash is about to run a CLI binary, the AI must load the matching skill first, before the command executes. The skill holds the WHEN/WHAT (auth, flags, error semantics, edge cases); the binary executes the HOW. Skipping the load step means flying blind on syntax.

CLI Skill auto-loaded Why
bun /bun Runtime + package manager — bun-specific APIs, scripts, lockfile.
gh /gh-cli, /git-flow-master Repo ops, PRs, gh api patterns + branch strategy.
git /git-flow-master Branching strategy detection, commit conventions, conflict resolution.
supabase /supabase, /supabase-postgres-best-practices DB CLI + Postgres patterns + DB scaffold flow.
vercel /vercel-cli, /deploy-to-vercel, /sprint-development Vercel CLI cookbook (verification, env sync, debug, rollback) + community deploy workflow + sprint-dev staging/prod deploy steps.
acli /acli Atlassian CLI — Jira/Confluence workflows, slug syntax, custom-field IDs.
playwright-cli /playwright-cli, /sprint-development Browser automation — sprint-dev E2E checks + standalone QA capture.
resend /resend-cli Transactional email CLI — send, templates, domains.
jq /acli JSON processor — required by acli to parse --json output.

The full mapping is the source of truth in CLAUDE.md § 6.5; the CLI names match exactly the EXTERNAL_CLIS constant in cli/install.ts. Today this is a rule-based gate (system prompt instruction); a future PreToolUse hook can enforce it.

7./sprint-development workflow

/sprint-development is the driver for ticket-led work. It runs the standard 12-step cycle: precheck the epic, transition the Jira ticket through dev states (Ready For Dev → In Progress → In Review → Ready For QA), plan, code, code review, deploy to staging, optionally deploy to production. For TDD slices it composes with /unit-testing mid-flight.

Example: "Add empty state to the user list when no results match the filter." Ticket exists, AC is 3 bullets, scope is one component plus one helper. /sprint-development drives the whole thing.

SDD bundle is no longer auto-installed

Earlier versions of this boilerplate auto-installed the /sdd-* bundle and let /sprint-development compose it for complex tickets. The installer has been switched to gentle-ai install --preset minimal, which installs only Engram (persistent memory). To use SDD, install the bundle manually with gentle-ai install --agent <agent> --components sdd,skills — then drive it standalone, outside /sprint-development.

Engram (persistent memory MCP) — auto-saves decisions, bugs, conventions; auto-recalls across sessions.

Skill composition — how skills load each other

Skills do not run in isolation. A workflow skill orchestrates an entire stage, but it delegates the actual I/O to tool skills that wrap a CLI or MCP. Tool skills, in turn, never invoke other workflow skills — they only run shell commands or MCP calls. This one-way layering keeps the orchestration graph acyclic.

1
Workflow skill (e.g. /sprint-development) — owns the stage, transitions Jira, plans the PR.
2
Tool skills (e.g. /git-flow-master, /acli, /playwright-cli, /gh-cli) — loaded just-in-time when Bash needs to run git, acli, playwright, or gh. Hold the syntax + flags + error semantics for their CLI.
3
Foundation reference host (agentic-dev-core) — passively cited by every workflow skill for shared doctrine (briefing template, dispatch patterns, model routing, TypeScript patterns). Never invoked directly.

Read top-to-bottom. Tool skills sit at the bottom of the food chain.

10.Commands & Scripts

One-shot utilities — neither orchestrate long flows like skills nor maintain config like setup scripts. You type them by name.

Commands

Single-file markdown at .claude/commands/<name>.md. Invoke with /<name>.

  • /sync-ai-memory Audit + patch the 5 always-included docs (README.md, CLAUDE.md, CONTEXT.md, INSTALLER.md, docs/agentic-development-engineering.md) plus the standalone HTML docs/onboarding.html and auto-detected high-frequency markdown. Preserves human-authored structure.
  • /business-data-map Generate / update .context/business/business-data-map.md — entities, flows, state machines, external integrations.
  • /business-feature-map Generate / update .context/business/business-feature-map.md — feature catalog by domain, CRUD matrix, API inventory, flags.
  • /business-api-map Generate / update .context/business/business-api-map.md — auth model, critical journeys, architecture, external integrations.
  • /master-implementation-plan Generate / update .context/master-implementation-plan.md — high-level dependency-cascaded roadmap of what to build first and why.

Scripts

TypeScript CLIs declared in package.json. Run with bun run <name>.

  • bun run setup Interactive installer — gentle-ai (Engram via --preset minimal) + 5 canonical MCPs (Tavily, Context7, Supabase, n8n, Atlassian) + external CLI verification (vercel, supabase, acli, playwright, resend).
  • bun run agents:setup Interactive walkthrough that fills .agents/project.yaml.
  • bun run vars:check Validate every {{VAR}}, {{jira.<slug>}}, and <<VAR>> reference resolves against the manifest + catalog.
  • bun run jira:sync-fields Catalog all custom fields from the live Jira workspace into .agents/jira-fields.json.
  • bun run jira:sync-issues Sync Jira issues (Story / Bug / Task / Epic) with their acceptance_criteria_gherkin field and other custom data into local context.
  • bun run jira:check Verify the Jira workspace satisfies .agents/jira-required.yaml.
  • bun run api:sync Sync OpenAPI spec from the live API and regenerate api/schemas/ types.
  • bun run up Update the boilerplate from upstream — pulls latest skills, scripts, and conventions while preserving local edits.
  • bun run lint:check / lint:fix Lint the codebase with ESLint. lint:fix auto-fixes.
  • bun run format:fix / format:check Format with Prettier. format:check only verifies.
  • bun run types:check Run tsc --noEmit against the whole project.
  • bun run onboarding Serve this onboarding page on a local port and open it in the browser.

Run bun <script> --help on any script for usage. The Vitest binary is installed locally — no global install needed.

Artifact refresh cycle

The five utility commands generate living artifacts — files that drift as the codebase evolves. Re-running them is normal maintenance, not a one-shot. Use this table to decide when to refresh each artifact.

Command Artifact Refresh when…
/sync-ai-memory README, CLAUDE.md, CONTEXT.md, INSTALLER.md, this page Major doc drift, new skill added, env/script renamed, repo restructured, onboarding flow changed.
/business-data-map .context/business/business-data-map.md Schema migration shipped, new entity, state machine changed, external integration added.
/business-feature-map .context/business/business-feature-map.md New feature merged, feature flag added/removed, CRUD endpoint added, domain reshuffled.
/business-api-map .context/business/business-api-map.md Auth model changed, critical journey added, API architecture decision made, new external service consumed.
/master-implementation-plan .context/master-implementation-plan.md Backlog reshaped, dependency cascade changed, roadmap pivot, sprint plan diverged from reality.

Rule of thumb

If a teammate (or your future self) would read the artifact and reach a wrong conclusion, it is stale — re-run. Otherwise, leave it. These commands are idempotent: re-running them with no drift produces a no-op diff.

11.Pieces you NEVER invoke directly

These are passive loads — the AI pulls them in as part of orchestration, not because you typed a command.

  • Tool / community skills (/playwright-cli, /gh-cli, /skill-creator, /find-skills, next-best-practices, tailwind-css-patterns, shadcn, etc.): the AI loads them automatically when a workflow skill needs to talk to a browser, GitHub, or a specific stack. You can force-load them when you genuinely need a one-shot operation, but they are usually invoked by other skills, not by you.
  • agentic-dev-core/references/* (briefing-template.md, dispatch-patterns.md, orchestration-doctrine.md): the AI loads these when a workflow skill delegates to a subagent. They are passive references — the subagent loads them inside its own context to know the canonical briefing format and dispatch decision rules.
  • Engram (persistent memory) — installed by gentle-ai, used proactively per CLAUDE.md §12.
  • Engram (MCP persistent memory): not a skill — an MCP server that auto-captures decisions, conventions, and bug fixes across sessions. See the dedicated callout below.

Engram — cross-session memory the AI manages for you

Engram is an MCP server that persists structured memory across sessions and compactions. The AI calls mem_save proactively (no user prompt needed) whenever it makes an architecture decision, fixes a bug, learns a convention, or hits a non-obvious gotcha. Subsequent sessions recall those memories via mem_search + mem_get_observation, so "we decided to use JWT instead of session cookies" survives without you having to re-explain it.

What the AI saves automatically: architecture / design decisions, conventions, bug root causes, non-obvious discoveries, workflow changes, user preferences, project-specific gotchas.

What the AI searches automatically: any message containing "remember", "recall", "what did we do", or any reference to past work — in any language you write in.

Session close: before saying "done", the AI calls mem_session_summary with Goal · Discoveries · Accomplished · Next Steps · Relevant Files. That summary is the seed for the next session. Skip it and the next session starts blind.

Full protocol: global CLAUDE.md § Engram. Installed by gentle-ai at user level — not committed in this repo.

12.MCPs available

The four canonical MCPs are configured by bun run setup. Each one fills a specific need:

MCP Use it for When NOT to use
Context7 Official library / framework docs (React, Next.js, Supabase, Tailwind, etc.) "How do I solve X" — that's Tavily territory
Tavily Web search, community solutions, troubleshooting Library API syntax — that's Context7
Supabase Database queries, schema inspection, migrations, project state App-level business logic — that's the application code
n8n Workflow automation, scheduled triggers, third-party integrations Synchronous user-facing API logic

Plus the always-available Engram MCP (persistent memory, installed by gentle-ai) — not in the table above because you don't invoke it; the AI calls it automatically on every save-worthy event.

Decision rule for the AI (and for you when deciding which to suggest):

  • Context7 for "how to use X" (official docs)
  • Tavily for "how to solve X" (community solutions)
  • Supabase for database / project state
  • n8n for workflow automation

13.TL;DR mental model

  • Skills = play scripts for the AI (multi-act, with characters that delegate).
  • Commands = single-act utilities you type by name.
  • Scripts (bun run …) = what YOU run in the terminal to keep the fuel (.agents/project.yaml + Jira fields) healthy.
  • .context/ = what the AI KNOWS about the project (generated by /project-foundation and the /business-*-map commands, kept fresh by /sync-ai-memory).
  • Application code = the real product, scaffolded by /project-bootstrap and built feature-by-feature via /sprint-development.

Fill project.yaml first (bun run agents:setup), then let the AI define the product with /project-foundation, then scaffold the Next.js + Supabase skeleton with /project-bootstrap. From there, /sprint-development drives every Jira ticket from Ready For Dev all the way to Ready For QA.
