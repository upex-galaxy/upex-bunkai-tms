## 2. Plan at a glance

This is the dev-side roadmap that turns the PRD/SRS into a launchable MVP and then keeps going. We ship **Bunkai Cloud MVP** first (Next.js 15 + Supabase + Vercel) covering 47 planned features across nine epics (BK-001 through BK-009). Once the design-partner cohort proves out the adoption metrics in `executive-summary.md ¬ß3`, we open into **Phase 2** ‚Äî ~14 features that take Bunkai from "validated SaaS" to "real open-core": Docker Compose self-hosted bundle, agentic execution protocol, CI/CD adapters, 2D mind-map, semantic search, Test Plans, configurable dashboards. **Phase 3** (8 features + 5 Enterprise gates) is the post-PMF horizon: 3D mind-map toggle, bidirectional Jira sync, SSO/SAML, compliance-grade audit log, native parameterization editors, marketplace.

The plan is organized into **7 MVP waves** (Wave 0 scaffolding ‚Üí Wave 7 launch hardening) followed by 4 Phase-2 waves (8‚Äì11) and a bullet-level sketch of Phase 3. Total MVP critical path: **Wave 0 ‚Üí 1 ‚Üí 2 ‚Üí 3 ‚Üí 4 ‚Üí 5 ‚Üí 7** with Wave 6 paralleling Waves 4‚Äì5 once Wave 3 is done. Rough estimate for a 2-engineer team: **12‚Äì18 weeks** ‚Äî speculative until Wave 0 closes, refine after `/project-bootstrap`.

The non-negotiable structural rule that orders every Wave is this: **no feature can land before the entities, RLS policies, and auth context it depends on**. This is why Wave 1 (Tenancy + Identity + RLS) blocks every other Wave ‚Äî a Project without a workspace cannot exist, an ATC without an RLS policy is a data leak waiting to happen.

## 3. Sequencing principles

The order below is derived from the entity dependency topology in `business-data-map.md ¬ß2` and the journey traces in `business-api-map.md ¬ß3`. It is not a preference ‚Äî it is what the schema forces.

1. **Tenancy + RLS first.** Every entity in the system carries `workspace_id` directly or transitively. Building Modules, ATCs, or Tests before the RLS policy that gates them is building a cross-tenant data leak. Wave 1 lands tenancy, membership, and the RLS pattern; everything downstream inherits from it.
2. **Hierarchy entities before content authoring.** Project ‚Üí Module is the substrate for User Stories, ATCs, Tests, and Bugs. The materialized `modules.path` column and the tree-rebuild query (FR-029) are the workhorse read every other screen depends on. Wave 2 establishes them.
3. **Content authoring before execution.** You cannot run a Test that does not exist. The chain US ‚Üí AC ‚Üí ATC ‚Üí Test must close before Wave 4 (Runs) can move. Wave 3 builds the ATC library (the differentiator).
4. **Execution after authoring.** Run ‚Üí run_atcs ‚Üí run_steps ‚Üí Bug. This is Journey 2 end-to-end. Wave 4 (Runs) and Wave 5 (Bugs) are sequential because Bug-from-Run is the canonical filing flow and depends on a failed `run_atc` being addressable.
5. **Views parallelizable once entities exist.** Tree view (FR-029), Table view (FR-030), command palette (FR-031) all read from already-existing data. Wave 6 batches them with the API-hardening work that needs no UI.
6. **API + CLI foundation in parallel with UI, gated on auth.** Bearer-PAT auth (FR-034) and the OpenAPI surface (FR-033) cannot start before Wave 1 (the auth model). But once Wave 1 is done, FEAT-040 / FEAT-041 / FEAT-042 / FEAT-043 can develop in parallel with later UI waves ‚Äî same endpoints, different consumer.
7. **Cross-cutting bolted on per-endpoint as features land, not big-bang.** Idempotency (FR-037), audit-light (FR-038), soft-delete (FR-039), Realtime (FR-040) are wired into each endpoint as part of that endpoint's Definition of Done ‚Äî not delivered as a separate cross-cutting wave. The infrastructure (the `idempotency_keys` table, the `activity_log` table, the `archived_at` columns, the Realtime channels) lands in Wave 0; the per-endpoint integration lands with the endpoint.
8. **Observability + NFR gates layered last, before launch.** Sentry, PostHog, rate-limit middleware, CSP, accessibility audit, and the NFR-validation pass (`non-functional-specs.md ¬ß1`) belong in Wave 7. Shipping them earlier means re-wiring after every Wave's surface changes.

## 5. What to implement first and why (per Wave)

One subsection per MVP Wave. Each rationale is product-facing (Why it matters) plus what is unblocked next (What it unblocks) plus the boundary conditions an experienced engineer would actually defend (How to scope it).

### Wave 0 ‚Äî Infrastructure scaffolding (the `/project-bootstrap` deliverable)

**Why it matters**: Without this, no feature lands. This wave is the output of `/project-bootstrap`, not implementation work in the product sense ‚Äî but it is the first sequencing gate.

**What it unblocks**: every Wave 1+. The auth shell, the schemas folder, the `bun run api:sync` pipeline, the error envelope helper, the `vars:check` linter, the OpenAPI Scalar UI route ‚Äî all become real here.

**Dependencies**: `.context/PRD/`, `.context/SRS/`, `DESIGN.md`, `.agents/project.yaml` (all present). The Discovery Gaps from this plan (¬ß12) feed `/project-bootstrap` Step 1 ‚Äî schema must close them before migrations are written.

**How to scope it**:
- Next.js 15 App Router + Tailwind v4 + shadcn/ui + Monaco + TanStack Table baseline.
- Supabase project provisioned (DB + Auth + Realtime + Storage). RLS template policy ready but not applied to specific tables yet.
- `app/api/v1/route.ts` shell + `api/schemas/*.types.ts` regenerated from `api-contracts.yaml`.
- DESIGN.md tokens lifted into `tailwind.config.ts` + CSS variables.
- `app/api/openapi.json` mounted (returns spec verbatim; no endpoints to introspect yet).
- `idempotency_keys`, `activity_log`, `feature_flags`, `user_view_state` tables created up-front so cross-cutting wiring is per-endpoint, not retrofit.
- `bun run vars:check` + `bun run api:sync` green; CI pipeline runs them.
- **Most likely scope creep to resist**: do not implement any FR-001..FR-040 logic here. This wave is shape, not feature work.

### Wave 1 ‚Äî Tenancy + Identity (EPIC-BK-001, FR-001..FR-004)

**Why it matters**: this is the safety net for every other feature. RLS policies that depend on `workspace_members` must be defined before any tenant-scoped table accepts writes. If Wave 1 ships with the wrong RLS pattern, every later Wave silently leaks data and we discover it under load.

**What it unblocks**: every other Wave. Supabase Auth + the cookie middleware + the `workspace_members` RBAC table + the invite-with-signed-token flow are the substrate Karim-style Bearer auth later layers onto.

**Dependencies**: Wave 0 (Supabase project, Next.js shell, Zod schemas).

**How to scope it**:
- Sign-up + sign-in (GitHub OAuth + Google OAuth + magic-link) via Supabase Auth ‚Äî `FR-001`.
- `POST /workspaces` creates `workspaces` + auto-inserts creator as `owner` in `workspace_members` ‚Äî `FR-002`.
- `POST /workspaces/{id}/invites` with signed 24h tokens ‚Äî `FR-003`.
- Workspace switch ‚Äî rotates `active_workspace_id` in session ‚Äî `FR-004`.
- **The RLS pattern**: write the canonical `EXISTS (SELECT 1 FROM workspace_members WHERE wm.workspace_id = <table>.workspace_id AND wm.user_id = auth.uid() AND wm.status = 'active')` policy here once, in one migration, and apply it to every `workspace_id`-bearing table as part of that table's Wave creation.
- **Most likely scope creep to resist**: SSO/SAML, custom role hierarchy, workspace deletion. All Phase 3 Enterprise. MVP runs on the four canonical roles.
- **Exit criteria**: founder and one invitee can both reach the empty Workspace Home, see only their own workspace's data, the activity log shows both joins. Cross-tenant query test (impersonating user B, querying workspace A's data) returns zero rows.

### Wave 2 ‚Äî Project & Hierarchy (EPIC-BK-002 + EPIC-BK-003, FR-005..FR-008)

**Why it matters**: the Project + Module tree is the substrate the entire UI sits on. The left-side navigation, defect heatmap rollup axis (`bugs.module_id`), Run filtering, and ATC location all depend on this hierarchy being correct.

**What it unblocks**: Wave 3 (ATCs need a Module + US to anchor to), Wave 4 (Runs filter by Module), Wave 5 (Bug heatmap rolls up by Module).

**Dependencies**: Wave 1 (workspace context + RLS pattern).

**How to scope it**:
- `POST /projects` + `GET /projects` ‚Äî workspace-scoped slug ‚Äî `FR-005`.
- `POST/PATCH/DELETE /modules` with recursive parent chain ‚Äî `FR-006`. Materialized `modules.path` column. Depth-‚â§6 hard guard + soft warning at depth 4. Circular-parent check returns `409 MODULE_CIRCULAR_PARENT`.
- `POST /user-stories` + `POST /acceptance-criteria` ‚Äî Markdown bodies ‚Äî `FR-007`, `FR-008`.
- **Minimal tree view (Wave-2 slice of FR-029)** ‚Äî Modules only, no ATC/Test/Run rollup yet. Pure recursive CTE. Status dots come in Wave 4.
- **Most likely scope creep to resist**: Jira import (`FR-009`) ‚Äî defer to Wave 3 alongside ATC library work. Bulk-edit, Table view ‚Äî Wave 6.
- **Exit criteria**: design-partner can create a 5-level deep Module tree with 10 US and 30 AC, see them in the tree view, navigate to a US and edit AC positions. Cross-tenant test still passes.

### Wave 3 ‚Äî ATC Library (EPIC-BK-004 + FR-009 Jira import, FR-010..FR-014)

**Why it matters**: this is **Bunkai's differentiator**. If Wave 3 ships well, the design-partner sees the "one-edit-many-tests" promise become real and the product's core value-prop is validated. If it ships badly, every other Wave is decoration on a generic CRUD.

**What it unblocks**: Wave 4 ‚Äî Tests are chains of ATCs, so without ATCs there are no Tests to run.

**Dependencies**: Wave 2 (Module + US + AC must exist before an ATC can anchor to them).

**How to scope it**:
- `POST /atcs` ‚Äî transactional insert of `atcs` + `atc_steps` + `atc_assertions` + `atc_acceptance_criteria` ‚Äî `FR-010`. The mandatory-AC link is enforced at the schema / Zod-validator level, not at the application layer.
- `GET /atcs?q=` ‚Äî Postgres `tsvector` GIN search on `title` + `tags` ‚Äî `FR-011`. Rank by `ts_rank` + recency decay.
- `PATCH /atcs/{id}` ‚Äî cascade-replace of `atc_steps` + `atc_assertions`, increment `version`, emit `atc.updated` with `affected_test_ids` ‚Äî `FR-012`.
- `GET /atcs/{id}?expand=used_in` ‚Äî usage report ‚Äî `FR-013`.
- `POST /atcs/{id}/duplicate` ‚Äî `FR-014`.
- `POST /imports/jira` + `GET /imports/{id}` ‚Äî async Jira import (ADF ‚Üí Markdown, AC heuristic extraction, dedup by `external_id`) ‚Äî `FR-009`. Resolves Discovery Gap G1 from data-map (the `imports` table is materialized here).
- **Most likely scope creep to resist**: semantic search via pgvector. Explicitly Phase 2 (`phase2.semantic_search` flag). Stay on tsvector for MVP.
- **Exit criteria**: 100 ATCs in <500ms p95 search latency (NFR target). Editing one ATC and viewing two Tests that reference it shows the edit propagated. A Jira import of 50 issues completes in <30s end-to-end (NFR ¬ß1 bulk operation budget).

### Wave 4 ‚Äî Tests + Manual Runs (EPIC-BK-005 + EPIC-BK-006, FR-015..FR-024)

**Why it matters**: this is the first wave where the design-partner does **the actual job** ‚Äî author a Test, run it, mark steps pass/fail. Every prior Wave is setup; this is the value loop. The Realtime row broadcast in `run_steps` updates is also the moment the dashboards stop being mockups and start being live.

**What it unblocks**: Wave 5 (Bug-from-Run drawer needs a failed `run_atc` to inherit `atc_id` from). Also unlocks executive-summary ¬ß3 success metrics (activation = workspace with ‚â•1 Run).

**Dependencies**: Wave 3 (Tests chain ATCs).

**How to scope it**:
- `POST /tests` + `PATCH /tests/{id}` ‚Äî Test creation + chain reorder (diff old vs new `atc_chain[]`, rebalance `position` integers in `test_steps`) ‚Äî `FR-015`, `FR-016`, `FR-018`.
- `GET /tests/{id}?expand=atcs.steps,atcs.assertions` ‚Äî the expanded view + the agent's single-roundtrip contract fetch ‚Äî `FR-017`.
- `POST /runs` ‚Äî transactional insert of `runs` header + full `run_atcs` and `run_steps` skeleton (all `pending`) ‚Äî `FR-019`. Idempotency-Key handling required from day one (don't retrofit).
- `POST /runs/{id}/steps/{stepId}/result` ‚Äî leaf update + derive parent `run_atcs.status` + recompute `runs.progress_pct` **in the same transaction** ‚Äî `FR-020`.
- `POST /runs/{id}/abort` + `POST /runs/{id}/finish` ‚Äî terminal-state transitions ‚Äî `FR-021`, `FR-024`.
- `GET /runs` ‚Äî Run history per Test + project-wide filter ‚Äî `FR-022`, `FR-023`.
- Realtime row subscriptions on `runs` / `run_atcs` / `run_steps` ‚Äî `FR-040`.
- The Run runner UI screen (focused mode with Pass/Fail/Block kbd shortcuts + Markdown notes + evidence URL paste).
- **Most likely scope creep to resist**: agentic-mode WebSocket protocol (Phase 2 ‚Äî `phase2.agentic_protocol`). The agent path in MVP uses the same REST endpoints + polling, not a separate channel.
- **Exit criteria**: 1 Test of 7 ATCs runs end-to-end with mixed pass/fail/block + evidence URLs + notes + Realtime updates visible in a second browser tab. Run history list shows the completed Run. Project-wide Run filter aggregates correctly across 3 environments.

### Wave 5 ‚Äî Bugs + Defect heatmap (EPIC-BK-007, FR-025..FR-028)

**Why it matters**: this is what makes Bunkai "a TMS, not a test case archive". Bugs filed from inside the Run, anchored to Module, rolling up into the heatmap within 5s ‚Äî this is the dashboard story the QA Lead persona (Mateo) needs to justify adoption to engineering management.

**What it unblocks**: Wave 7 launch hardening (the heatmap is one of the NFR validation targets; the Jira sync worker is a reliability surface that needs SLI numbers before GA).

**Dependencies**: Wave 4 ‚Äî Bug-from-Run drawer reads `module_id` from the failing `run_atc`'s parent ATC and `atc_id` from the failing `run_atc` itself.

**How to scope it**:
- `POST /bugs` ‚Äî file Bug + auto-link `atc_id` from `run_atc` when `run_id` set + refresh `module_defect_stats` materialized view in same transaction ‚Äî `FR-025`.
- `GET /bugs?module_id=&severity=&status=` ‚Äî listing with aggregates ‚Äî `FR-026`.
- `GET /projects/{id}/defect-heatmap` ‚Äî reads `module_defect_stats` MV ‚Äî `FR-027`.
- Async Jira-sync worker ‚Äî `FR-028`. Best-effort, never blocks bug creation. `jira-bug-sync-retry` cron handles failures with exponential backoff.
- Bug-from-Run side drawer (auto-populated `module_id`, `steps_to_reproduce` from executed steps, `atc_id`, `run_id`).
- **Most likely scope creep to resist**: bidirectional Jira sync (Phase 3 ‚Äî `phase2.jira_bidirectional` flag is named with Phase-2 prefix but ships Phase 3, see Discovery Gap ¬ß12-G8). Bug edit endpoints ‚Äî only status transitions in MVP.
- **Exit criteria**: a bug filed from inside a failing Run appears in the heatmap within 5s (NFR target). The same bug is filed-and-synced to a Jira sandbox project within 30s when the integration is enabled; if the Jira sandbox is unreachable, the bug remains usable in Bunkai natively and `jira_sync_status=failed`.

### Wave 6 ‚Äî Views + Search + API hardening (EPIC-BK-008 + EPIC-BK-009, FR-029..FR-036)

**Why it matters**: density and headless operation. Wave 6 is where Mateo gets the Table view to read trends and Karim gets the CLI to drive Bunkai from CI. Both unlock distinct adoption signals ‚Äî power users and AI-agent users.

**What it unblocks**: launch hardening (Wave 7) needs the full API surface live for the rate-limit + observability passes.

**Dependencies**: Wave 1 (auth for Bearer-PAT) + Wave 3 (entities to read in Table view). Wave 6 starts in parallel with Wave 4 once Wave 3 closes.

**How to scope it**:

- Persist view state per (user_id, project_id, view_kind) ‚Äî `FR-032`. Resolves Discovery Gap G3 from data-map (`user_view_state` table ‚Äî already scaffolded in Wave 0).

- **Exit criteria**: design-partner can drive a full Bunkai flow (create Project ‚Üí import US ‚Üí author ATC ‚Üí create Test ‚Üí start Run ‚Üí report results ‚Üí file Bug) via `bunkai` CLI **without touching the browser**. `/api/openapi.json` consumed by an external generated client without manual schema patching (executive-summary ¬ß3 success criterion).

## 6. Features / state machines that block others

Items where shipping them unblocks ‚â•2 other features, or shipping them late actively rots downstream work. Stack-ranked by leverage.

### 6.1 RLS policy pattern (Wave 1)

**Why the dependency is real**: every `workspace_id`-bearing table inherits this policy. Get it wrong once, every downstream Wave inherits the bug. The mitigation pattern in `architecture-specs.md ¬ß2.5` (indexed `workspace_id` + `SET LOCAL row_security = off` for trusted queries) is non-trivial ‚Äî designing it badly creates a performance ceiling.

**Downstream rework risk**: Wave 2 onwards. A bad RLS pattern surfaces as either cross-tenant leak (security) or per-query overhead (performance) ‚Äî both are migrations-level fixes after the fact.

**Smallest unblocking version**: one canonical policy expression, applied in Wave 1 to `workspaces` + `workspace_members` + `projects` (just to prove the pattern). Every later Wave applies the same pattern to its new tables.

**Partial-unblock for parallel work**: feature flag `SUPABASE_RLS_ENFORCED=true` in non-prod can be toggled off temporarily during local dev ‚Äî but every CI job runs with it on.

### 6.2 `modules.path` materialized column (Wave 2)

**Why the dependency is real**: every tree view, breadcrumb, subtree filter, and "list ATCs under this Module subtree" query reads this column. Recursive CTEs are unacceptable for the workhorse read at scale (NFR ¬ß1: tree view < 800ms p95 at 500 modules / 5000 ATCs).

**Downstream rework risk**: Waves 3 (ATC subtree search), 5 (Module-rooted defect heatmap), 6 (Tree view fully-loaded).

**Smallest unblocking version**: materialized column populated by a trigger on `modules` insert/update/parent-change. No fancy materialized view ‚Äî the column itself is the artifact.

**Partial-unblock**: not applicable. This is foundational, must land in Wave 2.

### 6.3 `executor.type` polymorphism on `runs` (Wave 4)

**Why the dependency is real**: the entire three-mode promise (manual / agent / CI) rests on dashboards aggregating without conditional logic. If `executor.type` is special-cased anywhere in the Run / Bug rollups, Phase 2 agentic + CI imports are not drop-in.

**Downstream rework risk**: Phase 2 Waves 9 + 10 (agentic + CI adapters). If MVP code paths special-case `human`, every Phase-2 import path becomes a re-test.

**Smallest unblocking version**: a single enum column + a single string `executor.identity`. Discovery Gap G7 (data-map): confirm whether `executor.identity` should be a single polymorphic string or split into `executor_user_id` (FK) + `executor_label` (string). Decision needed in Wave 0 or early Wave 4.

**Partial-unblock**: keep it polymorphic string in MVP; revisit only if Phase 2 stress reveals a real query problem.

### 6.4 Idempotency-Key infrastructure (Wave 0 table, applied per-endpoint Wave 1 onwards)

**Why the dependency is real**: every POST that creates non-trivial state (Workspace create, Project create, US create, ATC create, Test create, Run start, Bug file, Jira import trigger) is supposed to honor it (FR-037). Retrofit per-endpoint after the fact means rewriting every write handler.

**Downstream rework risk**: every Wave that adds a POST endpoint without honoring `Idempotency-Key` from day one will need a rewrite to be agent-safe.

**Smallest unblocking version**: middleware that checks the header before calling the handler + writes the response snapshot on the way back out. Land the middleware in Wave 0; apply it per-endpoint as endpoints land.

**Partial-unblock**: not applicable ‚Äî middleware is bolt-in, no half-version.

### 6.5 `module_defect_stats` materialized view (Wave 5)

**Why the dependency is real**: heatmap renders in <5s of new Bug (executive-summary ¬ß4 success criterion). Without the MV, the heatmap query becomes a runtime aggregate over `bugs` + `modules` recursive subtree ‚Äî slow at any non-toy data volume.

**Downstream rework risk**: Wave 7 NFR validation will reject the heatmap latency if this MV is not in place.

**Smallest unblocking version**: refresh-on-`bugs`-insert trigger + nightly safety-net cron (Wave 5 scope).

**Partial-unblock**: can ship Wave 5 without the trigger (manual refresh acceptable for first design-partner) and add the trigger before Wave 7.

## 7. Hidden dependencies

Features that share an entity, a state machine, an external integration, or a business rule but live in different epics in the feature-map. These are the silent integration costs.

### 7.1 `modules` shared by 5 different features

**Shared element**: the `modules` entity + its materialized `path` column.

**Features on each side**: FEAT-006 (Module CRUD), FEAT-016 (ATC search with module-subtree filter), FEAT-025 (Run filtered by Module), FEAT-032 (Bug listing by Module), FEAT-033 (defect heatmap rolling up by Module).

**What goes wrong if designed in isolation**: every consumer query joins on `modules.path` or recursively traverses `parent_module_id`. If the path-recompute trigger lags, every consumer sees stale data. If the subtree-filter query is not indexed for `(project_id, path)`, every Wave's filter is slow under load.

**Cheapest coordination point**: one engineer owns the `modules` migration + indexes in Wave 2. Every later Wave imports those indexes by reference, not by re-deriving them. Add a contract test ("subtree filter for /cart returns all descendants of /cart in <100ms with 500 modules") that runs in CI from Wave 2 onwards.

### 7.2 `bugs.module_id` couples Wave 5 + Wave 7

**Shared element**: the materialized view `module_defect_stats`.

**Features on each side**: FEAT-031 (File Bug from Run ‚Äî writes), FEAT-033 (defect heatmap ‚Äî reads), Wave 7 NFR validation (latency assertion against the heatmap endpoint).

**What goes wrong if designed in isolation**: filing a Bug + refreshing the MV inside the same transaction (Wave 5 design) versus refresh-async (a tempting late-Wave-7 perf optimization) changes the semantics of "heatmap updates within 5s of new Bug". Pick one and stick to it.

**Cheapest coordination point**: refresh-trigger lives in Wave 5; Wave 7 only adds the nightly safety-net cron. Document the choice in an ADR (ADR-007 candidate).

### 7.3 Realtime channels + Run runner + dashboard

**Shared element**: Supabase Realtime channels keyed by `project_id`.

**Features on each side**: FEAT-026 (step-result push to runner ‚Äî Wave 4), FEAT-031 (Bug filed ‚Üí heatmap refresh ‚Äî Wave 5), FEAT-047 (Realtime infrastructure ‚Äî applies to both).

**What goes wrong if designed in isolation**: each Wave subscribes to its own channel namespace. Project View tries to subscribe to `runs.project_id=X` but Wave 5 subscribed to `bugs.project_id=X` on a different topic. Result: one of the two UIs stops updating live.

**Cheapest coordination point**: one channel-naming convention documented in Wave 0 (`project:<project_id>:<table>`). Every consumer subscribes via the same helper.

### 7.4 `idempotency_keys` cross-cuts every write endpoint

**Shared element**: the `idempotency_keys` 24h-TTL table + the middleware that reads it.

**Features on each side**: FEAT-002, FEAT-005, FEAT-014, FEAT-021, FEAT-025, FEAT-031 (every POST that creates non-trivial state).

**What goes wrong if designed in isolation**: per-endpoint Idempotency-Key handling drifts (some endpoints honor it, some don't; some scope by `(endpoint, key)`, some by `(workspace_id, key)`). Karim persona breaks because retries either duplicate or 500.

**Cheapest coordination point**: middleware-level, not handler-level (see ¬ß6.4). Document the contract in `architecture-specs.md` ADR-005 (already planned).

### 7.5 Jira integration touches 3 features in 2 directions

**Shared element**: the `integrations` row (kind=jira, config + secrets_ref).

**Features on each side**: FEAT-011 (Jira import inbound, Wave 3), FEAT-034 (Bug ‚Üí Jira outbound, Wave 5), Phase 3 bidirectional sync.

**What goes wrong if designed in isolation**: inbound import worker (Wave 3) and outbound sync worker (Wave 5) reach into the same row to read credentials. If the secrets layout shifts between Waves, both break silently.

**Cheapest coordination point**: a single `getJiraClient(integrations_row)` helper module owned by one engineer. Both workers import it. Discovery Gap ¬ß12-G2 (integrations management API absent) must close before either worker ships.

### 7.6 Bearer-PAT + RBAC role coupling

**Shared element**: the auth context passed by middleware (`{ user_id, workspace_id, scopes }`).

**Features on each side**: FEAT-041 (Bearer-token auth, Wave 6) intersects with every Wave 1+ feature that does an RLS-gated write. The scope (`read|write`) is ANDed with the role (`viewer|member|admin|owner`).

**What goes wrong if designed in isolation**: handler guards check role but forget scope, or vice versa. A `member`-minted read-only PAT can write ATCs. Failure mode is a silent over-permission.

**Cheapest coordination point**: a single `requireScope('write')` guard helper exported from middleware. Every Wave-6 endpoint uses it.

## 10. Edge implementation cases

Features with ambiguous scope, pending technical decisions, or competing approaches. Grouped by theme.

### 10.1 Pending architectural decisions

- **Sync vs async for Jira sync** (FR-028). Decision recorded: async with retry, never blocks bug creation. Risk if violated: Bug filing latency held hostage by Jira availability.
- **In-process vs queue for run-timeout sweeper** (Discovery Gap ¬ß12-G5). MVP: Vercel cron every 15 min. Phase 2: BullMQ + Redis. Defer the decision by making the sweeper a serverless cron function so it's swappable.
- **Polling vs WebSocket for agent step-result reporting**. MVP: polling via REST. Phase 2: WebSocket (`phase2.agentic_protocol` flag). Lock-in mitigation = same REST endpoints honored regardless.
- **NestJS extract or stay-in-Next.js for service layer** (architecture-specs ¬ß3). Lock the decision: stay in Next.js for MVP. Revisit only when agentic-mode WebSocket + CI job load justifies it (Phase 2).

### 10.2 Scope-ambiguous features

- **Settings vs API surface for Integrations + Environments** (Discovery Gap ¬ß12-G2). MVP path: Server-Action-only, no public Bearer-protected REST. If Karim persona needs to manage these from CI, escalate to API surface in Wave 6.
- **Hard-delete admin endpoint scope**. MVP: only Modules support an admin hard-delete with confirmation header. All other entities are soft-delete-only. Don't expand without an explicit ask.
- **Workspace plan setter** (Discovery Gap ¬ß12-G7). MVP: manual SQL by founder for design-partners. Defer Stripe webhook to Phase 2.

### 10.3 Competing approaches

- **Build vs buy on search** ‚Äî Postgres `tsvector` GIN (build) vs Meilisearch / Typesense (buy). MVP: tsvector. Move only if p95 search latency degrades past 500ms (NFR ¬ß3 escape hatch).
- **shadcn copy-in vs Radix-direct for UI primitives** ‚Äî chosen: shadcn copies + Tailwind. Tradeoff: more code in our repo, full control. Phase 2 will not revisit this.
- **Token introspection** (Discovery Gap ¬ß12-G9) ‚Äî RFC 7662-style `/auth/introspect` vs the existing `GET /me` returning scopes. MVP: use `GET /me`. Phase 2 may add the formal introspection endpoint.

### 10.4 Permission / role boundaries

- **Token scope `write` is currently binary**. A `member`-minted token with `write` scope can create ATCs and file Bugs and start Runs ‚Äî all bundled. Phase 3 (Enterprise) may need finer-grained scopes (`atcs:write`, `runs:write`). Out-of-scope for MVP ‚Äî flag if a design-partner asks.
- **Custom role hierarchy** (Phase 3 `enterprise.role_hierarchy_custom`) ‚Äî out-of-scope for MVP. Hardcode the four canonical roles.
