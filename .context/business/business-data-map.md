## 1. Executive Summary

### What does this system do?

Bunkai is a **Test Management System (TMS)** built around a single non-negotiable structural rule: **every ATC (Atomic Test Component) must be anchored to a User Story AND at least one Acceptance Criterion**. Test cases are not free-form scripts вҖ” they are **ordered chains of reusable ATCs**, so editing one ATC propagates to every Test that chains it ("one-edit-many-tests"). Bugs live inside the test cycle (anchored to Module + ATC + Run), not delegated to Jira. The same data model serves three execution modes вҖ” **manual**, **agentic** (AI), **automated** (CI/CD adapters in Phase 2) вҖ” producing runs that aggregate transparently in dashboards.

Two editions ship from one codebase: **Bunkai Cloud** (Supabase-hosted) for the MVP, and **Bunkai Community** (self-hosted Docker Compose, Phase 2). The data model is identical across editions; only the infrastructure substrate differs.

### Main Actors

| Actor                                   | Description                                                                                                                                                       |
| --------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **QA Engineer** (Elena persona)         | Authors User Stories, AC, ATCs, Tests. Runs manual executions, files Bugs from inside the runner. Day-to-day power user.                                          |
| **QA Lead / Manager** (Mateo persona)   | Configures Projects + Modules + integrations. Watches dashboards: heatmaps, defect density, Run trends. Approves test plans.                                      |
| **Workspace Owner / Admin**             | Invites teammates, assigns roles, mints Personal Access Tokens, configures Jira/integrations.                                                                     |
| **AI Agent / CLI / CI** (Karim persona) | API consumer with a workspace-scoped Bearer token. Drives Runs end-to-end, reports step results, files Bugs вҖ” using the **same endpoints** a human UI would call. |
| **Viewer** (read-only stakeholder)      | PMs, dev leads, regulators who consume dashboards without writing.                                                                                                |

### Value Proposition

- For **QA engineers**: structural traceability is enforced by the data model вҖ” orphan tests cannot exist. Maintenance collapses to one ATC edit propagating everywhere.
- For **QA leads**: defect heatmaps and run-status rollups fall out automatically from the schema (no manual rollups, no spreadsheets).
- For **AI agents**: every state-changing UI action has a 1:1 REST endpoint with idempotency keys. The system is operable headless from day one.
- For **regulated industries**: identical data model in self-hosted edition removes the SaaS data-sovereignty objection.

### Key Relationships

The map has **one rule that organizes everything else**: every ATC carries `user_story_id` (mandatory) and is joined to вүҘ1 row in `atc_acceptance_criteria` (mandatory). This is the **structural traceability** the product enforces (`business-model.md` В§Value Propositions): you cannot create an ATC without telling the system which user story and which acceptance criterion it serves. The product's pedagogy is delivered through this rule, not through documentation.

Other relationships of note:

- **Tests reference ATCs, they don't embed them.** `test_steps.atc_id` is a foreign key, not a copy. When an ATC mutates (`{{PROJECT_KEY}}-012`), every Test that chains it sees the new version automatically вҖ” this is the "one-edit-many-tests" promise made tangible at the schema level.
- **Runs snapshot, they don't reference live.** `run_atcs` and `run_steps` are independent rows so historical Run integrity is preserved even if the underlying ATC is later edited. `run_atcs.atc_id` is a pointer for navigation; the step content executed at that moment lives durably in `run_steps`.
- **Bugs are anchored to Modules (mandatory).** This is what makes the defect heatmap (`{{PROJECT_KEY}}-027`) possible without runtime joins or per-tenant configuration: Module is the rollup axis, period.
- **Bugs optionally link to ATCs + Runs.** Filing a Bug from inside a Run (Journey 2, Step 4) auto-fills both вҖ” but a free-floating Bug filed from the Bugs view still works for "I saw something weird, ticket it now".
- **Modules form a tree via `parent_module_id` + materialized `path`.** `{{PROJECT_KEY}}-006` caps depth at 6 (soft warning at 4). Recursive CTEs power the Tree View; the materialized `path` column (`/cart/add-to-cart`) avoids recursion for breadcrumbs and subtree filters.

**Flow Narrative:**

1. Scheduler kicks the agent process. Agent loads its PAT from secrets and calls `GET /me` to validate.
2. Middleware verifies the token via SHA-256 hash lookup in `access_tokens`, checks expiry + revocation, attaches workspace context.
3. Agent fetches the full Test contract in one request (`?expand=...`) вҖ” ordered ATCs with their steps and assertions вҖ” so it knows the entire script up front.
4. Agent calls `POST /runs` with an `idempotency_key` (typically the run's logical name like `2026-05-19-nightly`). If the agent retries due to a transient failure, the same key returns the original `run_id` ({{PROJECT_KEY}}-037, `idempotency_keys` table).
5. Per step: agent drives the application via Playwright, captures evidence to R2, then `POST .../result`. The transaction is **identical** to the manual flow вҖ” same endpoint, same processing ({{PROJECT_KEY}}-020 recompute parent statuses).
6. On a failure, agent fires `POST /bugs` with the same payload shape a human would submit. Bug appears in the same dashboards.
7. Agent calls `POST .../finish` with terminal status. Run is closed.

**Business Rules:**

- **Data shape is identical to manual mode.** This is the structural decision behind Bunkai's "three-mode" claim вҖ” dashboards aggregate without conditional logic. `executor.type вҲҲ {human, agent, ci}` is metadata, not a switch.
- **Idempotency is mandatory for safe retry.** Agents are expected to send `Idempotency-Key` on every create endpoint ({{PROJECT_KEY}}-037).
- **A `running` Run abandoned by an agent does not auto-close.** A sweeper cron (configurable `max_duration`, default 4h вҖ” Journey 3 alt-paths) marks abandoned Runs as `aborted` with reason "agent timeout".
- **Token scopes can be tighter than role.** A `member` user can mint a read-only token; the bearer of that token cannot create ATCs even though the underlying user could (architecture В§5 RBAC).

**Code Involved:**

- `app/api/v1/auth/tokens/route.ts` вҖ” {{PROJECT_KEY}}-034 issuance
- `middleware.ts` вҖ” Bearer verification + workspace-context attachment
- Same `/runs`, `/bugs` endpoints as Flow 2 (this is the point)
- Future sweeper cron: `app/api/cron/run-timeout/route.ts` (Vercel cron-scheduled)

**Side effects:** Same as Flow 2. Plus: each authenticated request increments a `last_used_at` on `access_tokens` (per {{PROJECT_KEY}}-034 token-usage semantics, implementation tracked in `/project-bootstrap`).

## 7. Multi-tenancy & Authorization (RLS)

Bunkai is a hard multi-tenant system at the database layer. Every entity that contains business data carries a `workspace_id` (directly or via a parent), and **Postgres Row Level Security** policies gate access at the row level rather than the application layer.

**Policy pattern** (architecture-specs.md В§2.5):

```
For every table with workspace_id:
  USING (
    EXISTS (
      SELECT 1 FROM workspace_members wm
      WHERE wm.workspace_id = <table>.workspace_id
        AND wm.user_id      = auth.uid()
        AND wm.status       = 'active'
    )
  )
```

**Role inheritance** (`viewer вҠӮ member вҠӮ admin вҠӮ owner`):

| Role     | Capabilities                                                         |
| -------- | -------------------------------------------------------------------- |
| `viewer` | Read-only across every table in the workspace                        |
| `member` | Full CRUD on Project entities (Modules, US, AC, ATC, Test, Run, Bug) |
| `admin`  | + Manages members, integrations, access tokens                       |
| `owner`  | + Workspace deletion, billing (Cloud plan)                           |

**Token scopes are an additional restriction on top of role** вҖ” a `member` user can mint a read-only PAT; that bearer cannot create ATCs even though the underlying user could.

**Performance mitigation** (founder conversation, architecture В§2.5): RLS overhead becomes measurable at scale. Mitigation = indexed `workspace_id` on every table + `SET LOCAL row_security = off` for trusted service-role queries (migrations, scheduled jobs) that already pre-filter explicitly.

## 8. Edition Variance (Cloud vs Community)

**The data model is identical across editions.** Only the infrastructure substrate differs. This is a deliberate decision: it preserves the "self-hosted is the same product" promise made in `business-model.md` В§Value Propositions and removes the data-sovereignty objection for regulated buyers.

| Layer           | Bunkai Cloud (MVP)                                  | Bunkai Community (Phase 2)                           |
| --------------- | --------------------------------------------------- | ---------------------------------------------------- |
| Database        | Supabase Postgres 16                                | Plain Postgres 16 in Docker                          |
| Auth            | Supabase Auth (JWT + OAuth)                         | Better Auth (or Auth.js) in-process                  |
| Realtime        | Supabase Realtime (logical replication вҶ’ WebSocket) | Redis pub/sub + custom WebSocket (extracted process) |
| Object store    | Cloudflare R2 (S3-compatible, egress-free)          | MinIO in Docker (S3-compatible)                      |
| Background jobs | Vercel cron + serverless functions                  | BullMQ + Redis                                       |
| Search          | Postgres `tsvector` GIN                             | Same                                                 |
| Errors          | Sentry SaaS                                         | Sentry self-hosted (community plan)                  |
| Analytics       | PostHog Cloud                                       | PostHog self-hosted                                  |

**What this means for data-model work**: any schema change, RLS policy, materialized view, or trigger written today must be expressible on plain Postgres without Supabase-specific functions. Supabase-specific surfaces (Auth, Realtime, Storage) are wrapped behind thin abstractions per architecture В§3 so that Phase 2 can swap implementations without touching business logic.
