## Welcome to Bunkai

Bunkai (分解 — "the art of breaking down a kata into its applications") is an **open-core Test Management System**. The name is a Japanese martial-arts term, not the anime word: surface that clarification anytime the brand is introduced. Behind the name sits a single conviction — that existing TMSs (Xray, Zephyr Scale, TestRail, qTest) are document vaults with execution glued on top, and that *the data model itself can teach good QA engineering* if it refuses to let bad QA happen.

Why does this matter? Because in every traditional TMS, the same step ("navigate to /login") is duplicated across forty test cases. A URL change is a forty-file edit. Traceability between User Story, Acceptance Criterion, Test, Run and Bug is a free-text field nobody fills. Reports show "80% passing" without telling you what those 80% actually validate. Bugs are dumped into Jira the moment something fails and the QA loop loses context within a week. Bunkai's bet is that **structural rules baked into the schema** — not better documentation, not better training, not better dashboards on top of the same broken model — are what flip the curve.

The structural rule that organizes everything else: **every ATC (Atomic Test Component) is mandatorily anchored to a User Story AND at least one Acceptance Criterion**. Tests are not free-form scripts — they are **ordered chains of reusable ATCs**, so editing one ATC propagates to every Test that chains it. The system rejects orphan ATCs at the constraint layer, not at the application layer. That's the moat. Everything else in this codebase exists to make that rule pleasant to live with.

- **Mateo Silva (QA Lead / Manager, 44, Madrid, regulated industry)** owns coverage, dashboards, and the awkward auditor conversation. His pain: he cannot answer "what does sprint X cover?" in under one minute with data anybody trusts. He is skeptical of "AI magic" but pragmatic — he'll adopt anything that demonstrably shortens his report-prep time.
- **Sara Iglesias (Full-Stack Developer, 29, Lisbon)** is the adjacent consumer. She opens Bunkai occasionally to check whether her PR breaks coverage, to read bug context, to link a feature back to tests. Her pain is context-switching tax: Jira, Xray, GitHub, Slack, console logs, repeat. She is fluent in Markdown, OpenAPI, basic SQL, and uses Claude Code in her editor.
- **Karim (Autonomous AI Test Agent — non-human)** is the surprise persona. Karim is not a person; Karim is a Playwright-driven Claude Code script, or a bespoke quality-engineering agent, or a CI job. Karim is what makes the "three execution modes share one data shape" claim real.

Elena and Mateo are the buyers / champions (Cloud and Enterprise respectively). Sara is the adoption multiplier — developers who appreciate the tool spread it across teams. Karim is the long-tail differentiator: every AI-driven workflow built on top of Bunkai's API increases switching cost away.

## Core mental model — the data spine

This is the single most important section in this guide. If you internalize the chain below, every other decision in the codebase becomes self-explaining.

```
Workspace ──► Project ──► Module (tree, depth ≤6)
                              │
                              ▼
                         User Story
                              │
                              ▼
                  Acceptance Criterion (1..N per US)
                              │
                              │ M:N
                              ▼
                            ATC  ←──── steps + assertions (ordered children)
                              ▲
                              │ test_steps (the chain — references, not copies)
                              │
                            Test
                              │
                              ▼
                            Run  ──► run_atcs ──► run_steps
                              │       (per-ATC      (per-step result,
                              │        rollup,       evidence, duration)
                              │        derived)
                              ▼
                            Bug  (anchored to Module mandatory,
                                  to ATC + Run optional)
```

Walk it from top to bottom.

A **Workspace** is the tenant root. Every business row in the system carries a `workspace_id` (directly or via a parent), and Postgres Row Level Security gates access at the row layer. One workspace = one team / customer. Inside a workspace sit **Projects** — one project per app-under-test. A workspace can host many products.

Inside a Project, **Modules** form a tree (depth capped at 6, soft warning at 4). Modules are *first-class entities*, not folders — defect heatmaps and coverage rollups happen by Module. A materialized `path` column (`/cart/add-to-cart`) means breadcrumbs and subtree filters do not need recursive joins at read time.

A **User Story** anchors testing to product intent (Markdown body, optional `external_id` → Jira issue key). Each US has 1..N **Acceptance Criteria** — atomic, sortable, Markdown-bodied, each one representing one testable behavior. This is the provenance unit. ATCs link to ACs, not to US directly — that's what makes "one ATC can satisfy multiple ACs" and "one AC can be covered by multiple ATCs" work as a clean M:N.

An **ATC** (Atomic Test Component) is *the heart of the product*. It carries: title, layer enum (`UI | API | Unit`), tags, an ordered set of **atc_steps** (content + input + expected), an ordered set of **atc_assertions**, and an `atc_acceptance_criteria` M:N join binding it to ≥1 AC of its parent US. **The data model rejects orphan ATCs at constraint level**. You cannot create an ATC without telling the system which User Story it serves and which Acceptance Criterion it satisfies. This is the product's pedagogy, delivered through schema rather than documentation.

A **Test** is a named, taggable container that owns an **ordered chain of ATC references** stored in `test_steps` (test_id + atc_id + position). The chain stores *references*, not copies. When you edit ATC-001, every Test that chains it sees the new version automatically — this is the "one-edit-many-tests" promise made tangible at the schema layer. **Tests are the unit of execution; ATCs are the unit of authorship.** This decoupling is the reason maintenance collapses to one place.

A **Run** is one execution instance of a Test against an environment. The header (`runs`) carries executor identity (human / agent / ci), idempotency key (24h TTL), progress percentage, status, and lifecycle timestamps. Below the header sit two tables: `run_atcs` (per-ATC rollup, status derived from children) and `run_steps` (per-step result with evidence URL, duration, notes, error message). Runs **snapshot, they don't reference live** — if an ATC is edited later, historical Run integrity survives because the step content that executed at the moment of the Run lives durably in `run_steps`.

A **Bug** is anchored to a Module (mandatory — this is what makes the defect heatmap possible without runtime joins or per-tenant configuration), optionally to an ATC + Run (auto-filled when filed from inside the runner), with severity P1–P4 and four-state lifecycle (open → in_progress → resolved → closed; reopen allowed). Bugs are *native*, not delegated to Jira. Jira sync is best-effort, asynchronous, and optional — when configured, the workflow worker creates a Jira issue and writes back the `external_id` + `external_url`.

If you remember nothing else from this guide: **no orphan ATCs, no monolithic tests, no bugs without modules**.

## Three execution modes — one data shape

Bunkai supports three execution modes — **manual**, **agentic** (AI-driven), **automated** (CI/CD adapters in Phase 2) — and the design decision that makes the product coherent is that **all three share the same `runs` + `run_atcs` + `run_steps` shape**. Dashboards aggregate without conditional logic. The only difference between a human Run and an agent Run is metadata: `executor.type ∈ {human, agent, ci}`.

Concretely: when Elena clicks "Pass" on step 3, the request is `POST /api/v1/runs/{run_id}/steps/{step_id}/result`. When Karim's Playwright script reports the same step pass, the request is **the same endpoint, the same payload shape, the same transaction**. The route handler updates `run_steps`, recomputes the parent `run_atcs.status` (derived value — clients never write it), recomputes `runs.progress_pct`, fires `activity_log`, and broadcasts a Realtime row change to every connected tab. Identical code path. The fact that you can build dashboards that aggregate human + agent + CI runs in one query without `CASE WHEN executor_type = ...` is the structural payoff.

This is the design decision behind the "three-mode" claim in the marketing copy. It is also the architectural reason the API surface lives in MVP scope (EPIC-BK-009) and not in Phase 2 — without API-first runs, agentic mode would require a parallel data model.

## Editions

Two editions ship from one codebase. A third tier is license-gated.

| Edition | Ships | Distribution | Substrate |
|---|---|---|---|
| **Bunkai Cloud** | MVP | hosted on Vercel + Supabase | Supabase Postgres, Auth, Realtime, Storage; Cloudflare R2; Vercel Cron |
| **Bunkai Community** | Phase 2 | self-hosted via Docker Compose, Apache 2.0 / MIT (TBD) | plain Postgres, Better Auth, Redis pub/sub + custom WebSocket, MinIO, BullMQ |
| **Bunkai Enterprise** | Phase 3 | Cloud OR self-hosted, commercial license | Cloud / Community + SSO/SAML, audit log, role hierarchy, priority support |

The data model is identical across all three. License gating happens via `workspaces.plan ∈ {community, cloud, enterprise}` plus a `feature_flags` table (workspace- or project-scoped boolean toggles). Phase-2 and Phase-3 features ship "dark" with the MVP infrastructure so enabling them later is a config change, not a deploy.

**What this means in practice when you write code today**: any schema change, RLS policy, materialized view, or trigger must be expressible on plain Postgres without Supabase-specific functions. Supabase-specific surfaces (Auth, Realtime, Storage) live behind thin abstractions so the Community-edition swap is contained. Do not embed `auth.uid()` calls or Supabase client APIs in business logic that will eventually run on Better Auth.

## Tech stack at a glance

| Layer | Bunkai Cloud (MVP) | Community Phase-2 substitute |
|---|---|---|
| Frontend | Next.js 15 App Router + React 19 + TypeScript | same Next.js image in Docker behind a reverse proxy |
| Styling | Tailwind + shadcn/ui + Radix + Lucide icons | unchanged |
| Editor | Monaco for ATC step authoring | unchanged |
| Table | TanStack Table | unchanged |
| API | Next.js Route Handlers (`app/api/v1/...`) | same; extract NestJS when WebSocket + job-queue load justifies it |
| Validation | Zod (client + server, shared schemas) | unchanged |
| Database | Supabase Postgres 16 | plain Postgres 16 (`postgres:16-alpine`) in Docker |
| Auth | Supabase Auth (JWT + OAuth GitHub/Google + magic-link) | Better Auth or Auth.js in-process |
| Realtime | Supabase Realtime (logical replication → WebSocket) | Redis pub/sub + custom WebSocket |
| Object store | Cloudflare R2 (S3-compatible, egress-free) | MinIO in Docker (S3-compatible) |
| Background jobs | Vercel Cron + serverless functions | BullMQ + Redis |
| Search | Postgres `tsvector` GIN | same; `pgvector` opens semantic search in Phase 2 |
| Errors | Sentry SaaS | Sentry self-hosted |
| Analytics | PostHog Cloud | PostHog self-hosted |
| CI/CD | GitHub Actions + Vercel preview deploys | unchanged |

Why this stack? Three reasons. First, Next.js 15 App Router with RSC keeps the JS bundle under our LCP/TTI budgets on the Vercel free tier from cold cache. Second, Supabase bundles four of our top integration surfaces (Postgres, Auth, Realtime, Storage), shrinking MVP iteration time materially. Third, every Supabase surface has a credible self-hostable substitute that swaps behind a thin abstraction — the Community edition is not a rewrite, it's a substrate swap.

What lives **outside** the stack: the API is REST + OpenAPI 3.1 served at `/api/openapi.json`. The CLI is a Bun-compiled binary plus `npx bunkai`. Personal Access Tokens carry the prefix `bk_pat_` and are SHA-256 hashed at rest. The data model is the only durable artifact — every other layer is replaceable.

## Local development

> **Honest disclosure**: this repo is currently **foundation-only**. `package.json` at root contains the project-starter toolkit (CLI bootstrapping, lint scripts, agent setup), not the Next.js app yet. The Next.js scaffold, Supabase migrations, and runtime scripts (`dev`, `build`, `test`) land when `/project-bootstrap` runs. Until then the "local development" loop is documentation work, not running the product.

To get oriented today:

1. Clone the repo.
2. Run `bun install`.
3. Copy `.env.example` to `.env` and fill the variables (Supabase URL, anon key, service role key, Jira credentials, R2 keys, Sentry DSN, PostHog key — every required key is annotated in `.env.example`). **Never hardcode credentials and never commit `.env`.** Critical Rule #1.
4. Run `bun run vars:check` — confirms every project-variable placeholder referenced in skills and docs resolves cleanly against `.agents/project.yaml`.
5. Read `package.json` directly when you need to know which scripts exist. Do **not** trust this guide or `CLAUDE.md` to enumerate them — Critical Rule #10. As of this writing, the script inventory is foundation-tooling only: `setup`, `setup:doctor`, `agents:setup`, `up`, `onboarding`, `api:sync`, `vars:check`, `skills:check`, `skills:registry`, `jira:sync-*`, `jira:check`, `format:fix/check`, `lint:fix/check`, `types:check`, `repo:check`, `repo:fix`, `clean`. The runtime scripts (`dev`, `build`, `test:unit`, `test:e2e`, `db:migrate`, `supabase:start`) will be added by `/project-bootstrap`.

To start a real dev loop **after `/project-bootstrap`**, the canonical sequence will be: `bun install` → `bun run supabase:start` (local Supabase stack) → `bun run db:migrate` → `bun run dev` (Next.js on port 3000). Notes on architectural intent:

- **`app/api/v1/...`** is the authoritative URL surface — every state-changing UI action has a 1:1 Route Handler. The OpenAPI spec at `/api/openapi.json` is generated from these.

- **`lib/services/`** holds business logic. Route handlers stay thin (validate → call service → shape response). Services are testable without HTTP.
- **`lib/db/`** wraps the Supabase client behind thin factories so the Community-edition Phase-2 swap (Better Auth + plain pg) does not ripple into business logic.
- **`db/migrations/`** is the durable schema artifact. The ERD in `architecture-specs.md §2` is the contract; the migration files are the implementation. Forward-only, backward-compatible — Critical Rule #5 spirit.

## Conventions

These are the rules every contributor (human and AI) follows. The full set lives in `CLAUDE.md` §1 (Critical Rules) and `CLAUDE.md` §10 (TypeScript Stack Quick-Reference). Below is the narrative version.

**Plan before coding.** Produce an implementation plan (`implementation-plan.md` under the relevant `.context/PBI/{module}/{ticket}/` folder) before writing code. Flow is always Plan → Code → Review. This is Critical Rule #2.

**Read before edit.** Always read a file before editing it. Preserve formatting and indentation. Never overwrite without reading first. Critical Rule #7.

**Credentials live in `.env`.** Never hardcode, never guess, never commit secrets. Critical Rule #1.

**TypeScript discipline.** Strict mode (`"strict": true`, `noUncheckedIndexedAccess`, `noImplicitOverride`, `exactOptionalPropertyTypes`). No `any` without an inline justification comment. No `@ts-ignore` — use `@ts-expect-error` so it fails when the underlying issue is fixed.

**Function signatures: max 2 positional parameters.** Three or more parameters become an object param. This keeps call sites readable and refactor-safe.

**Imports use aliases (`@api/`, `@schemas/`, `@lib/`, `@components/`).** No deep relative imports (`../../../../`). The aliases are defined in `tsconfig.json`.

**Error handling is context-dependent.** Public-facing methods (route handlers, service entry points) **fail fast** — throw with a typed error class. Internal utilities **silent-fail** — return `null` or a `Result<T, Error>` so the caller decides. Never silently catch and swallow.

**Validation is at the boundary.** Every API request validated against a Zod schema. Every database response validated against a Zod schema if it crosses a trust boundary. Validation never happens "later".

**No domain coupling in `lib/utils/`.** Utilities are framework-agnostic — no React, no Next, no Supabase-specific APIs. Domain logic stays inside its feature folder. Move to `shared/` only when ≥2 features import the same code AND the abstraction is stable.

**Surgical changes.** Touch only what the task requires. Match existing style even if you'd write it differently. Don't refactor unbroken code as a side effect. Notice unrelated dead code? Mention it; don't delete it without an explicit ticket.

**Commits look human-authored.** Never include "Generated with Claude Code", "Co-Authored-By: Claude", or any AI attribution. Critical Rule #3.

**Never push to `main` without explicit confirmation.** Critical Rule #4. Never rewrite pushed history, never force-push to shared branches. Critical Rule #5.

**Scripts: read `package.json` directly.** Do not quote scripts from `CLAUDE.md`, this guide, or any other documentation. Drift kills. Critical Rule #10.

## First-day playbook

If you just joined Bunkai and want to be productive in one work day, do these in order:
